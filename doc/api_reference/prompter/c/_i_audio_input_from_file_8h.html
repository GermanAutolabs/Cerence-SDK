Detailed Description

Interface of the 'AudioInputFromFile' audio module, for DDFW only.

This interface is DEPRECATED and for DDFW only. SDK users shall use the AudioFromFile interface.

In general the AudioInputFromFile module offers the possibility to stream audio data from an audio file list with 1 to n elements to subsequent connected audio modules. The streaming is automatically started and stopped when an audio session becomes active respectively inactive. For each audio session one file is streamed only. Each start of a session triggers the AudioInputFromFile audio module to stream the next audio file from the file list. In case the file list would exceed it is streamed from the beginning again.

The AudioInputFromFile module is usable in two manifestations:

1. AUTOMATICALLY CREATED WITH A PRE-CONFIGURED AUDIO FILE LIST IN THE AUDIO CONFIGURATION FILE (.json): With calling prepareScenario() resp. activateScenario() an involved AudioInputFromFile instance is automatically created. NOTE: In this use case the audio_files parameter in the audio configuration file (.json) is mandatory! With automated creation there is no interface handle available to set a file list by customer API.

2. CREATED BY THE CUSTOMER WITH THE OPTION TO SET AN AUDIO FILE LIST DYNAMICALLY BY THE API: Use the following customer API to create and set up an AudioInputFromFile module. NOTE: In this use case the audio_files parameter in the audio configuration file (.json) is optional in case it is set by the customer API before an audio session is started!

Supported file formats are:

  • standard wav files
  • NIST files
  • header-less raw files, it has to be kept in mind, that multichannel signals are expected in interleaved format (123123123123 and not 111122223333).
  • OGG/OPUS files
  • encoded audio files with a constant bitrate (blocksize needs to be configured as 'samples_per_channel' in the audio format)

The audio file path of each element in the audio file list (pre-configured or set with customer API) is build using the following mechanism:

1. A file with an absolute path remains unchanged. A set root path is not considered. Example:

  • audio_files="/test_path/test_file.wav" ==> "/test_path/test_file.wav"

2. A file with a relative path is prepended with the pre-configured resp. set root path. The resulting path is expected to be in relation to the JSON configuration directory. Examples (JSON config dir = "../config/json"):

  • root_path="" + audio_files="test_file.raw" ==> "../config/json/test_file.raw"
  • root_path="test_path" + audio_files="test_file.raw" ==> "../config/json/test_path/test_file.raw"
  • root_path="test_path" + audio_files="../test_file.raw" ==> "../config/json/test_file.raw"
