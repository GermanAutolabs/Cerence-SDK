Scope Of Document

The main scope of this SDK documentation is to serve as user guide for engineers who use Cerence SDK to develop and deliver project-specific applications.

SDK CloudServices Overview

The CloudServices component provides connectivity between the SDK and Cerence Cloud Services (NCS) servers. It is used internally by Prompter and Recognizer components if they are configured to use cloud.

In addition it can also be used to directly communicate with the NCS server. Therefore three diffent interface-modules are provided: CloudCommander, CloudDataUploader, CloudLogger. These modules internally are all based on a common protocol implementation called CloudServices.

At the moment only HTTP protocol is supported by the SDK:



Note

It is recommended to use Prompter and Recognizer APIs for cloud-prompting and cloud-recognition. CloudServices should only be used for scenarios not covered by those APIs (e.g. controlling server-dialogs).

For cloud-connectivity there is always a set of server-credentials needed which is not shipped with this SDK.

In the following sections we describe the basic usage and concepts of the CloudServices API, the full API documentation and header files for the CloudServices are included in SDK release packages.

Document conventions

API call examples in this document are shown in C syntax.

CloudManager

The CloudManager is a central management instance responsible for creation and controlling of all CloudServices instances.

Creating CloudManager

To create a CloudManager instance, you may use the follow API:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudManager_create
(
  const LH_CHAR *name,
  nuance_common_ISystemManager *systemManager,
  const nuance_common_IConfiguration *config,
  nuance_cloudservices_ICloudManager **cloudManager
);

Configuring CloudManager

The configuration of the CloudManager itself is optional. By configuration it is possible to enable cloud-logging to NCS servers. For details see the Logging section.

As the CloudManager is responsible for the creation of all CloudServices instances it must receive their configuration in the passed IConfiguration object.

The CloudServices configuration defines the protocol that is used for client-server communication and the URL of the server to be used. It must also contain the credentials for client-authentication and all parameters of the NCS protocol which are generic over all commands. Optionally the audio-format / encoding that is used for the client-server transfer can be configured. It is strictly recommended to stay with the default audio-format unless there are good reasons for changing it.

The JSON snippet example below shows how to define one CloudServices instance called CloudServices_0.

{
  "version": "CloudServices 1.0",

  "cloud_services" : [{
    "name" : "CloudServices_0",
    "type" : "http",
    "url" : "https://sample-url.nuancemobility.net:443/NmspServlet/",
    "app_id" : "SAMPLE_APP_ID",
    "app_key" : "xxxxxxxxxxxxx",
    "uid" : "sample_user_id",
    "device_model" : "sample_device_model",
    "ssl_certificate_bundle" : "./ca-bundle.crt",
    "app_name" : "CDFW_Sample",
    "app_version" : "1.0",
    "app_language" : "en-us",
    "carrier" : "n/a"
  }]
}

CloudCommander

The CloudCommander is a module that allows to send any NCS command to the NCS server.

Configuring CloudCommander

As a prerequisite to be able to use one or more CloudCommander instance(s) in an application, the desired instances must be defined in the JSON configuration.

The JSON snippet example below shows how to define two CloudCommander instances appCmdr and appCmdr2. Both use different CloudServices instances for cloud connectivity. Using different CloudServices instances allows parallel sending of multiple commands. Using the same CloudServices with multiple CloudCommanders will serialize all cloud commands and would be similar to only using one CloudCommander.

{
  "version": "CloudServices 1.0",

  "cloud_module_config": [{
    "type": "CloudCommander",
    "name": "appCmdr",
    "cloud_services_name": "CloudServices_0"
  },
  {
    "type": "CloudCommander",
    "name": "appCmdr2",
    "cloud_services_name": "CloudServices_1"
  }]
}

See chapter JSON configuration to find a brief overview of the CloudCommander JSON API.

Creating CloudCommander

To create a CloudCommander instance, you may use the follow API:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudCommander_create
(
  const LH_CHAR *name,
  nuance_audio_IAudioManager *audioManager,
  nuance_cloudservices_ICloudManager *cloudManager,
  const nuance_common_IConfiguration *config,
  nuance_cloudservices_ICloudCommanderListener *moduleListener,
  nuance_cloudservices_ICloudCommander **cloudCommander
);

The instance name for each CloudCommander must be unique. While the CloudManager handle is mandatory, the AudioManager handle is only needed if audio needs to be streamed from / to the NCS cloud.

Using CloudCommander

There are two different ways to use the CloudCommander. You can either send out commands that are pre-configured in a JSON configuration file, or you can use the programmatic API to build up the CloudCommand manually. While the first one is more convenient, the second one is much more powerfull.

Sending pre-defined commands

Configuring cloud commands

As a prerequisite to be able to send a cloud command it needs to be configured in the JSON configuration file. Each command-configuration consists of of the command-name from the NCS specification and a command-specific dictionary. In addition a list of parameters can be provided. The following parameter-types are currently supported:

  • STRING
  • DICTIONARY
  • SEQUENCE.

AUDIO and TTS parameters are not supported by this API, instead the propmter and recognizer API should be used to trigger such commands.

{
  "version": "CloudServices 1.0",

  "cloud_command": [{
    "name": "CerenceNluCancel",
    "timeout": 10000,
    "command": "DRAGON_NLU_APPSERVER_CMD",
    "dictionary": {
      "dictation_type" : "ccpoi_assistant",
      "dictation_language" : "eng-USA",
      "application":"dragondrive_eval_1.0"
    },
    "params":[{
      "param_name": "REQUEST_INFO",
      "type": "DICTIONARY",
      "dictionary": {
        "start" : 0,
        "end" : 0,
        "text" : "",
        "appserver_data" : {
          "message" : "cancel",
          "time" : "2012-11-09T11:45:15-0500",
          "timezone" : "America/Toronto",
          "speaker" : "user",
          "error_on_user_data_not_uploaded" : "0",
          "type" : "conversation",
          "actions" : []
        },
        "binary_results" : 0
      }
    }]
  }]
}
Sending cloud commands

The CloudCommander API should be used with the method nuance_cloudservices_ICloudCommander_sendCloudCommand, e.g.:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudCommander_sendCloudCommand
(
  nuance_cloudservices_ICloudCommander *instance,
  const LH_CHAR *commandName,
  nuance_cloudservices_ICloudCommandJsonListener *listener,
  nuance_cloudservices_ICloudCommand **cloudCommand
);

In this method the name of the command-configuration can be passed. The sendCloudCommand method is implemented asynchronous. It directly returns an ICloudCommand instance and starts the transfer to the server in the background. The ICloudCommand inherits from the nuance.common.IBackgroundOperation and therefore can be used to wait until the command is completed or to abort the execution of the command. Usually, an application will wait until a command has been completed, before continuing with further processing. This can be achieved by calling the synchronous method waitForCompletion after sendCloudCommand, e.g.:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudCommand_waitForCompletion
(
    nuance_cloudservices_ICloudCommand *instance,
    nuance_common_IBackgroundOperation_Status *status
);

If the application needs to cancel the execution of a command, this can be done with the call nuance_cloudservices_ICoudCommand_abortOperation, e.g.:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudCommand_abortOperation(nuance_cloudservices_ICloudCommand *instance);

The CloudCommand API provides a listener interface that calls the applications’s callback functions asynchronously to notify the application when internal errors or events occurred. If using the nuance_cloudservices_ICloudCommander_sendCloudCommand API, a listener will be provided by the API and will received various events.

The following events are defined:

typedef void (*FUNC_nuance_cloudservices_ICloudCommandJsonListener_onCommandStarted)
(
    void *instance,
    const nuance_cloudservices_ICloudCommand *cmd
);

typedef void (*FUNC_nuance_cloudservices_ICloudCommandJsonListener_onCommandFinished)
(
    void *instance,
    const nuance_cloudservices_ICloudCommand *cmd
);

typedef void (*FUNC_nuance_cloudservices_ICloudCommandJsonListener_onCommandResult)
(
    void *instance,
    const LH_CHAR *result,
    LH_BOOL isComplete,
    const nuance_cloudservices_ICloudCommand *cmd
);

typedef void (*FUNC_nuance_cloudservices_ICloudCommandJsonListener_onCommandError)
(
    void *instance,
    const LH_CHAR *error,
    const nuance_cloudservices_ICloudCommand *cmd
);

Both results and errors are delivered as JSON-formatted strings.

Create cloud commands

Instead of defining a cloud-command in the JSON configuration the programmatic API can be used to build up a cloud-command manually. This API has no restrictions regarding audio- and TTS- parameters like the configuration based API. Nevertheless Prompter and Recognizer APIs should be preferred for those scenarios.

The CloudCommander API should be used with the method nuance_cloudservices_ICloudCommander_createCloudCommand, e.g.:

CLOUDSERVICES_C_EXPORT nuance_common_ResultCode nuance_cloudservices_ICloudCommander_createCloudCommand
(
  nuance_cloudservices_ICloudCommander *instance,
  const LH_CHAR *commandName,
  LH_S32 commandTimeout,
  nuance_common_IDictionary *optionalSettings,
  nuance_cloudservices_ICloudCommandListener *listener,
  nuance_cloudservices_ICloudCommand **cloudCommand
);

Similar to the attributes of the JSON configuration the equivalent runtime parameters are provided to configure the command. A command created programmatically supports all typed parameters, namely STRING, DICTIONARY, SEQUENCE, BYTES, AUDIO and TTS. Based on the NCS specification, multiple command parameters can be added to the created command to accomplish the desired cloud request. Once all the necessary parameters are added, the command should be finished by calling nuance_cloudservices_ICloudCommand_finish(), which in term signals the server to process the command and send back the results. Please note that even though the command is finished, the audio data associated with the AUDIO and TTS parameters is still being streamed in real-time. The command is terminated upon the final result or an error is delivered to the user via the command listener.

For details see the CloudCommander API reference.

CloudDataUploader

The CloudDataUploader is a low-level module for data-upload (e.g. address-books) to the NCS cloud. It is recommended to use the DynamicContentConsumer API of the ASR component instead.

For details on this module see the CloudDataUploader API reference.

CloudLogger

The CloudLogger is a low-level module for sending log-messages to the NCS servers. A CloudLogger instance can be created either manually or automatically by enabling the cloud-logging in the JSON-configuration of the CloudManager.

For details on this module see the CloudLogger API reference.

JSON Configuration

The SDK module can be configured through JSON configuration files. These files have to be a part of the configuration files passed to the SDK service providing the cloud functionality. Multiple SDK services are able to share the same CloudServices configuration or use a dedicated one. Sharing the same configuration does not mean sharing a CloudServices instance. On the contrary each SDK Service will create its own CloudServices instance (or multiple of them).

The following sections give a formal specification of the JSON configuration format for the CloudServices module.

CloudServices

The input audio format.

This configuration key defines the audio format which is expected to be coming in from the Audio Backbone and then send to the cloud recognizer.

From that format also the configuration of the server connection is derived.

For cloud asr the highest ogg opus quality is achieved using “OPUS_16K”, which is activated using the following format:

  • “container”: “CONTAINER_OGG”
  • “encoding”: “ENCODING_OPUS”
  • “sample_rate”: 16000
  • “samples_per_channel”: 512
  • “buffer_count”: 1

For more details about supported audio formats please have a look at the “Supported codecs and audio encodings” section in the “Cerence Cloud Services HTTP Services 2.0 Developers Guide”.

"output_audio_format":

The output audio format.

This configuration key defines the audio format which will be send to the Audio Backbone after receiving it from the cloud tts.

From that format also the configuration of the server connection is derived.

For cloud TTS the best ogg opus quality is achieved using “OPUS_24K”, which is activated using the following format:

  • “container”: “CONTAINER_OGG”
  • “encoding”: “ENCODING_OPUS”
  • “sample_rate”: 24000
  • “samples_per_channel”: 2048
  • “buffer_count”: 1

For more details about supported audio formats please have a look at the “Supported codecs and audio encodings” section in the “Cerence Cloud Services HTTP Services 2.0 Developers Guide”.

Please be aware that the server originally synthesizes 22050Hz data, which is upsampled at the server due to the constrains of the opus codec. Therefore it is safe to downsample the data to 22050Hz without audible quality loss.

Nevertheless any downsampling below 22050Hz is not recommended since it will lead to audible quality losses.

"idle_session_timeout":
Specifies the maximum time, in seconds, that the Cloud Services instance can remain connected while idle. It is considered to be idle when no transactions are sent or replies are received. Cloud Services will automatically reconnect upon sending a new transaction. Must be greater than zero. Default is 45.
"connection_timeout":
Specifies the maximum time, in seconds, between the connection request (either at Cloud Services creation, or upon sending a transaction) and the connected confirmation is received from the server. Must be greater than zero. Default is 30.
"ssl_certificate_bundle":

An absolute or relative path to a file that holds a bundle of one or more certificates. The relative path is considered to be in relation to the JSON configuration directory

For any client connecting to Cerence Cloud it is mandatory to run server authentication (Cerence Corporate Security request). This means that during the TLS handshake the client (here: CloudServices) has to receive and validate the server certificates.

During establishment of a secure connection (e.g. HTTPS), an authenticity check of the remote peer’s certificate is performed. This way it is proved that the remote peer is really the one it claims to be. Server certificates are validated against the trust chain known to client. When using HTTPS the check is typically performed based on the CA (Certificate Authority) bundle located at the system’s default path. This property allows to change the path to a different user-specific certificate bundle.

The certificate bundle must contain a chain to the root certificates used to issue the Cerence Cloud server certificate. Note that root certificates in a trust chain may be withdrawn by the issuing authorities for security reasons. Therefore, it must be possible that applications can update the certificate bundle in the field during system lifetime.

Logging

Log communication between SDK client and NCS server

For debugging purposes the communication between the SDK client and the Cerence cloud can be traced by configuring corresponding log consumers,

Overview

For debugging purposes it may be necessary to understand what is exactly going on between the client and the NCS cloud. When using HTTP CloudServices there is an option to enable the debug-functionality of the underlaying CURL library.

The Logger offers two data logging points one for the transfer sent to the cloud, one for the data received from the cloud. They can be activated by defining a log consumer with zone LOG_DATA and the correct log-module name.

This document will describe how to identify and configure those log-points correctly.

General Advice

The logger configuration parameter "data_buffer_size_per_channel" should be set to a quite high value (e.g. “data_buffer_size_per_channel”:500000,) to avoid buffer overruns (cf. section “Buffer overrun”).

Available Logging Points

As already mentioned there are two dedicated logging points, one for the incoming and one for the outgoing data. Both are located in the CURL wrapping layer which defines the prefix for the log module name: “nuance.cloudservices.EasyCurlAdapter”. As the logging points are instance-specific they need to be extended by the instance name and an “in” or “out” suffix for the direction.

It is recommended to define one dedicated destination/file for each data logging point. This is done by writing the full definition of the logging point to the “modules” section of the log consumer configuration and using LOG_DATA as logging zone.

Finding names

When HTTP CloudServices are used directly the instance-name is the one passed during creation. It can also be found in the “cloud_services” section of the JSON configuration.

When using CloudServices integrated into a high level component the instance-name is inherited from the instance-name of this component.

Limitations

When logging to file the files are created when writing the first data package. If no data is transfered no file will be created.

Configuration

The following example shows how to configure/access those HTTP transaction logging points. It assumes that the instance name of HTTP CloudServices is “httpInst”. It can be used as a blueprint for setting up the desired configuration:

{
  "version": "Common 1.0",
  "logger": {
      "data_buffer_size_per_channel":500000,
      "consumers": [
          {
              "name":    "csHttp in transaction log",
              "output":  "csHttp_in_%DATE_TIME%.log",
              "modules": "nuance.cloudservices.EasyCurlAdapter.httpInst.in",
              "zones":   "LOG_DATA"
          },
          {
              "name":    "csHttp out transaction log",
              "output":  "csHttp_out_%DATE_TIME%.log",
              "modules": "nuance.cloudservices.EasyCurlAdapter.httpInst.out",
              "zones":   "LOG_DATA"
          }
      ]
  }
}

Send log data from SDK client to NCS server

SDK client applications can send log data to NCS servers using the ILogging_logText API, see Common user guide log text.

When an application calls nuance_common_ILogging_logText the log data handed over to this function will be sent by the SDK to NCS with a NVC_LOG_REVISION_CMD if:

  • the log zone is set to NUANCE_COMMON_LOGZONE_LOG_CLOUD
  • the cloud logging is enabled in the JSON configuration of the "cloud_manager", e.g.
{
"version": "CloudServices 1.0",
"cloud_manager":
 {
  "enable_cloud_logging": ["csOpusOggHttp", "csHttp"]
 }
}

Given such a configuration, the cloud manager will create the CloudServices instances needed for this logging automatically. By default, cloud logging is disabled. That means, if an application wants to use cloud logging, then "enable_cloud_logging" must be explicitly set for the corresponding CloudService instances.
