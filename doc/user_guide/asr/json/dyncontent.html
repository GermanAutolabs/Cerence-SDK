Dynamic Content Consumer Configuration

OVERVIEW

Dynamic Content Consumers (DCCs) are used to process lists of dynamic data. For example, a phone book or music collection would be such a list. Depending on the type of dynamic content consumer, they do different kinds of processing. In this document, we show the available types of dynamic content consumers and show their respective configuration options. We also show how to connect consumers to applications.

COMMON OPTIONS

Every dynamic content consumer is built up following this schema:

{
  "version" : "VoConHigh 5.0",
  "dynamic_content_consumer" : [{
    "name" : "name",
    "content" : [{
        "type" : "<type>",
        "name" : "content name"
      }
    ],
    "type" : "<typeName>",
    "<typeName>" : {
      ...
    }
  }
  ]
}
name
This is the name of this content consumer. It is used when creating it from the API.
content

This is a list of fields that are part of this consumer.

type
The type of this content field. Can either be string, string_list, string_list_list, integer (32 bit integer), integer64 (64 bit integer), base64 or json.
name
The name of this content field.
type
The type of this consumer.
<typeName>
Holds the type specific configuration of the dynamic content consumer. Can either be list, or slm_fm.

LIST DYNAMIC CONTENT CONSUMER

list dynamic content consumers are used to compile lists of data into contexts usable by the on-board recognizer. Processing happens on the embedded system itself and no data is transferred to an off-board system.

Configuration Options

The following shows a sample configuration:

{
  "version" : "VoConHigh 5.0",
  "dynamic_content_consumer" : {
    "name" : "VAD_NAMES",
    "content" : [{
        "type" : "string",
        "name" : "first_name"
      },{
        "type" : "string",
        "name" : "last_name"
      }
    ],
    "type" : "list",
    "list" : {
      "lexicon" : {
        "uses" : "ENU_ML_PRON_CHAIN"
      },
      "search" : [{
          "name" : "vad_firstname_lastname",
          "acmod" : "acmod4_900_enu_gen_car_f16_v2_0_0.dat",
          "link" : [[{
                "type" : "content",
                "content" : {
                  "add_tag": "true",
                  "clc_category" : "normal",
                  "name" : "first_name"
                }
              },{
                "type" : "filler",
                "filler" : {
                  "clc_category" : "normal",
                  "value" : [
                    "of", "in the house of"
                  ],
                  "optional" : "true"
                }
              },{
                "type" : "content",
                "content" : {
                  "tag_name": "some_tag_name",
                  "clc_category" : "normal",
                  "name" : "last_name"
                }
              }
            ]
          ]
        },{
          "name" : "vad_firstname",
          "acmod" : "acmod4_900_enu_gen_car_f16_v2_0_0.dat",
          "link" : [[{
                "type" : "content",
                "content" : {
                  "tag_name": "some_tag_name",
                  "clc_category" : "normal",
                  "name" : "first_name"
                }
              }
            ]
          ]
        }
      ]
    }
  }
}

The list object contains the following keys.

lexicon
This specifies the lexicon to be used. Here, we reference a lexicon specified in another file. The details of the lexicon specification are given in the lexicon configuration documentation.
search

The contexts specify in what way data from the content list is combined. It does so by specifying contexts with references to the content fields.

name
The name of the search. This has to be unique inside this specific DCC. It is used as reference by the applications to access the search
tag_name
A tag to use for all entries for this search. If the property tag_name is specified in search and content, the priority goes to the one at the search level. An exception is made for the leading and trailing content of a link arrow, in this case the priority will goes to the “tag_name” from xothe content block. This exception was introduced mainly to accomodate the title entry from RefDesign.
acmod
The acoustic model to use for generating this search.
link

Field links to the content list.

type
can be content or filler.
content
gives the clc_category to use for the lookup and the name of the referenced field, the key “tag_name” gives you the option to put the content of the link inside a tag. Tags are visible in the recognition result. They appear in the same way as tags/slot in a compiled bnf grammar.
filler
gives a list of possibly optional filler words (value) that can be uttered between content words, as well as the clc_category

Phonetic entries

In order to directly enter phonetic transcriptions for terminals the list DCC supports the following mechanism:

"content" : [{
    "type" : "string",
    "name" : "<terminal name>_Ortho"
  }, {
    "type" : "string_list",
    "name" : "<terminal name>_Pho"
  }
],

When a content of type “string” is named with the extension “_Ortho” and a content of type “string_list” is named with the extension “_Pho”, the list DCC takes the phonetic transcriptions for the orthography from the given string list of the corresponding terminal name. If no corresponding phoneme content is given the phonetic transcriptions are looked up in the lexicon.

Note that this naming convention is compatible with the COP file format, so COP files can directly be used as input for phonetic transcriptions.

It is possible to add a content field of type “integer” with the name “Entry_ID” to specify IDs for terminals:

"content": [{
    "type": "string",
    "name": "FirstName_Ortho"
  }, {
    "type": "string",
    "name": "LastName_Ortho"
  }, {
    "type": "integer",
    "name": "Entry_ID"
  }
],

In addition, the list DCC also supports per-terminal IDs. When enrolling terminals from content fields of type “string” that end in “_Ortho” (e.g. LastName_Ortho), the system will look for similar fields of type “integer” ending in “_ID” (e.g. LastName_ID). If any is found in the content list, the numbers will be associated and displayed in the recognition result as part of the _userID.lo32 and _userID.hi32 JSON fields:

"content" : [{
    "type" : "string",
    "name" : "LastName_Ortho"
  }, {
    "type" : "integer",
    "name" : "LastName_ID"
  },{
    "type" : "string",
    "name" : "FirstName_Ortho"
  },{
    "type" : "integer",
    "name" : "FirstName_ID"
  }
],

If no such fields are found, the system will continue to use the first integer field to continue to provide global row IDs (attached to the last or only terminal of the entry).

When linking to such contents in the DCC configuration, the postfix should be omitted from the link name as demonstrated below:

"link" : [[{
      "type" : "content",
      "content" : {
        "clc_category" : "normal",
        "name" : "FirstName"
      }
    },{
      "type" : "content",
      "content" : {
        "clc_category" : "normal",
        "name" : "LastName"
      }
    }
  ]
]

Note that it is not mandatory for the content to contain the orthography columns, however in that case either an entry ID or per-terminal IDs hast to be provided. Otherwise, the configuration will be considered as invalid.

Transcription Identifier entries

For using phonetic transcriptions from identifier dictionaries via transcription identifier the list DCC supports the following mechanism:

"content" : [{
    "type" : "string",
    "name" : "<terminal name>_Ortho"
  }, {
    "type" : "integer64",
    "name" : "<terminal name>_TransID"
  }
],

When a content of type “string” is named with the extension “_Ortho” and a content of type “integer64” is named with the extension “_TransID”, the list DCC takes the phonetic transcriptions for the orthography from any given identifier dictionary. More information about the identifier dictionary can be found in the lexicon configuration documentation. If no corresponding phoneme content is given the phonetic transcriptions are looked up in the lexicon.

Note that this naming convention is compatible with the COP file format, so COP files can directly be used as input for transcription identifiers.

Connecting to an Application

To connect these dynamically generated contexts to an application, we simply define a dynamic context entry and specify the DCC name and the context name we want to use:

{
  "version" : "VoConHigh 5.0",
  "application" : [{
    "input" : [{
      "name" : "UC_A1M",
      "type" : "asr",
      "asr" : {
        "search" : [{
            "name" : "Gate_BNF",
            "type": "static",
            "static": {
              "filename" : "Gate_BNF.fcf"
            }
          }, {
            "name" : "VAD_SLM",
            "type": "static",
            "static": {
              "filename" : "VAD_SLM.fcf"
            },
            "use_highest_guest_accuracy_on_host" : 0,
            "update_all_computed_params" : 1,
            "slot_filling" : [{
                "slot" : "DragonDriveDemoSlm#firstLastNameList",
                "search" : {
                  "type": "dynamic",
                  "dynamic": {
                    "dynamic_content_consumer" : "VAD_NAMES",
                    "name": "vad_firstname_lastname"
                  }
                }
              }, {
                "slot" : "DragonDriveDemoSlm#firstNameList",
                "search" : {
                  "type": "dynamic",
                  "dynamic": {
                    "dynamic_content_consumer" : "VAD_NAMES",
                    "name": "vad_firstname"
                  }
                }
              },
              ...
            ]
          },
          ...
        ],
        ...
      }
    }]
    }
  ]
}

SLM-FM DYNAMIC CONTENT CONSUMER

Slm-FM match dynamic content consumer is used to compile lists of data into slm context, or fuzzy match contexts, usable by the on-board recognizer. Processing happens on the embedded system itself and no data is transferred to an off-board system. In other terms, SLM-FM DYNAMIC CONTENT CONSUMER is similar to LIST DYNAMIC CONTENT CONSUMER except the produced context is slm context, or fuzzy match context. It is important to note that for now, the contexts are always archived on disk.

Please note that since the created fuzzy match context consists of normalized strings, the result from this fuzzy match context will be normalized too. If “start_rule” is configured, then there will be no warning message about setting the start rule name in the logs. Otherwise, a warning message will be presented.

The fm_search object contains all keys in LIST DYNAMIC CONTENT CONSUMER’s list object, the difference consist of the context object. The fuzzy match context contains the following extra keys:

extra_word

This object is used to add extra words to the context. It has the following keys:

type : This is a mandatory key specifying how extra words will be added. It can be list or ewf.

list : words are passed in the same json config file. ex.:

"extra_word": {
  "type": "list",
  "list": [{
    "domain": "MUSIC",
    "category": "constraints",
    "orthography": "next"
  },
  {
    "domain": "",
    "category": "nlu",
    "orthography": "play"
  }]
}

ewf : words are passed through an encoded file format. ex.:

"extra_word" : {
  "type": "ewf",
  "ewf": {
    "filename": "extra_words_music.ewf"
  }
}
column

This specifies the fuzzy match context column. It has the following keys:

type : can be integer or string. This is a mandatory key and only one of the following type is allowed at a time:

integer : has the following options:

activatable : optional key indicating whether the column is activatable or not. The default falue is false.

accumulate_result: optional key indicating whether to accumulate result or not. The default value is false.

array: Indicate if the column is an array. The default value is false.

unique_values: Optional key indicating that the values in the column are unique. This is useful for the engine to know how to best create the activation information. Even if the flag is set, there is no check for the uniqueness of the values of that column. If the column is used for activation, only one instance of the value will be used in case there are duplicated values.

string : has the following options:

speakable : this block is specific to speakable column and it is optional to a string type.

clc_category: mandatory key indicating the clc category, (ex. artist).

activatable : optional key indicating whether the column is activatable or not. The default falue is false.

accumulate_result: optional key indicating whether to accumulate result or not. The default value is false.

array: Indicate if the column is an array. The default value is false.

In addition, the link key in the column object optionally supports an array of objects, to enable the combination of multiple content inputs. If a link array containing more than one content object is used, all content objects must be of type string. In this case, the values will be concatenated in order (separated by a whitespace) and then treated as a single column. Further, any related “_Pho” content will be ignored. If such an entry is to be treated as a single terminal, the split field must be set to false. Otherwise, default splitting will be applied.

Note that if a place holder term whose phonetic transcription is defined in a dictionary (e.g. SpeechContactRelation_Spouse) is needed to be used as a value for a column of an entry, this term can only be used alone and cannot be surrounded by other terms.

The slm_search object contains all keys in LIST DYNAMIC CONTENT CONSUMER’s list object, the difference consist of the context object. The fuzzy match context contains the following extra keys:

link

This specifies a list of content for dynamic content consumer context. It has the following keys:

type : can be content or filler. This is a mandatory key determine the one of below keys existence:

content : Required if and only if “type”: “content”, has the following keys:

name : The name of the referenced field.

clc_category : The Clc category to use for the lookup.

tag_name : tag name to add around the content of the link. Set this key if you want a specific tag to be added around the content of the link. These tags appear in the recognition result in the same way as other tags/slots (e.g. !tag(..) statement in bnf grammar)

filler : Required if and only if “type”: “filler”, has the following keys:

clc_category : The Clc category to use for the lookup.

optional : whether or not using optional filler words.

value : a list of possibly “optional” filler words.

Connecting to an application

Similar to LIST DYNAMIC CONTENT CONSUMER.

64 bit IDs

It is possible to define ID columns as 64 bit integer by using base64 or integer64 type as the content type. Column type remains as integer:

"content" : {
  "type" : "base64",
  "name" : "city_id"
}

"column": {
  "type": "integer",
  "name": "city_id",
  "integer": {
    "activatable": "true",
    "accumulate_result": "true"
  },
  "link": {
    "name": "city_id"
  }
}

The provided values should be encoded in “base64” or “integer64”.

Remark: The name “entry_id” is a reserved name and cannot be used as 64 bit integer field name (it can only be 32 bit integer). However, users are allowed to define their own 64 bit integer fields to use as entry ids by using a different name.

Phonetic entries

In order to directly enter phonetic transcriptions for terminals the fuzzy match DCC supports the following mechanism:

"content" : [{
    "type" : "string",
    "name" : "<terminal name 1>_Ortho"
  }, {
    "type" : "string_list",
    "name" : "<terminal name 1>_Pho"
  }, {
    "type" : "string_list",
    "name" : "<terminal name 2>_Ortho"
  }, {
    "type" : "string_list_list",
    "name" : "<terminal name 2>_Pho"
  }
],

When a “name” of type “string” has the extension “_Ortho” and a “name” of type “string_list” has the extension “_Pho”, the Fuzzy Match DCC takes the phonetic transcriptions for the orthography from the given string list of the corresponding terminal name. If no corresponding phoneme content is given the phonetic transcriptions are looked up in the lexicon.

The above applies likewise to combinations of type “string_list” for the orthographies and type “string_list_list” for the corresponding phonetic transcriptions.

Remark: This naming convention is compatible with the COP file format. So COP files can directly be used as input for phonetic transcriptions. Example of phonetic entry via COP file:

#ECO V1.2 UTF-8;

#ECOScheme COP1 V1.6
#EDBDatabaseID NBT_AddressBook_v_1_0
#EDBTableUpdateID AddressBookTable_20100708_0501
#EDBTableUpdateMode GivenInColumn
#EDBTableUpdateType FullUpdate

# Columns:
#Column1 Entry_ID
#Column2 EDBTableUpdateMode
#Column3 Application_Ortho
#Column4 FirstName_Ortho
#Column5 FirstName_Pho
#Column6 LastName_Ortho
#Column7 LastName_Pho

#Format(Entry_ID) Ident
#NumIDFormat(Entry_ID) Decimal
#NumIDBinLength(Entry_ID) 64

#Format(EDBTableUpdateMode) Ident

#SplitCharSequence(FirstName_Ortho) "\\;"


1 ; a;  "VAD"; "Heather\\;Jane"; "'hE.D$R+*#'d&Ze&In 'Z@%~"; "Morris" ; "'mO.R+Is 'mA.R+Is$"

REMOTE DYNAMIC CONTENT CONSUMER

Remote dynamic content consumers are used to feed data to off-board systems. This is in particular interesting if, for instance, the local phone book needs to be made available to an off-board system, such that the names from that phone book can be used in off-board recognition or dialogs.

Please note that it is not possible to define two dynamic content consumers in parallel.

Options

The following shows an example of a remote dynamic content consumer:

{
  "version" : "VoConHigh 5.0",
  "dynamic_content_consumer" : {
    "name" : "CONTACTS",
    "content" : [{
        "name" : "first_name",
        "type" : "string"
      }, {
        "name" : "additional_info",
        "type" : "json"
      }
    ],
    "type" : "remote",
    "remote" : {
      "query" : {
        "name" : "remote lm query",
        "command" : "DRAGON_NLU_DATA_UPLOAD_CMD",
        "connection_info" : {
          "uses" : "DRAGON_CONNECTION"
        }
      },
      "credentials" : {
        "uses" : "DRAGON_host"
      }
    }
  }
}

The remote object contains the following keys.

query

This object represents a query to a remote service.

name
Name of this query (for internal use only)
command
The command to execute
connection_info
The connection info object to use. Connection info is a plain named json object as described in the verbatim json objects documentation.
credentials
The credentials to use. Credentials are plain named json object as well.

Connecting to an application

Remote dynamic content consumers are not connected to application directly. Rather, they can be used to upload data to a specific server prior to executing an application that uses this server afterwards.

BAG OF WORDS DYNAMIC CONTENT CONSUMER

The bag of words dynamic content consumer implements the bag of words grammar modeling. The words of each added sentence are put into one BNF terminal disregarding any order.

Configuration Options

The following shows the configuration for the DCC type bag_of_words:

"bag_of_words": {
  "lexicon": {
    <lexicon configuration>
  },
  "search": {
    "name" : "<context name>"
    "acmod" : "<acousic model filename>",
  }
}

The bag_of_words object contains the following keys:

lexicon
The lexicon used to lookup the word transcriptions. Details of the lexicon specification can be found in lexicon configuration documentation.
search
The context configuration determines the name and the acoustic model of the generated context.

Bag Of Words grammar

A bag of words content value consists of a sentence and can contain a user ID. When a user ID is available it is associated to each string of the sentence in the grammar. Otherwise the grammar will not contain any ID.

The following content value input:

a; 1; "<string 11> <string 12> <string 13> <...>" ;
a; 2; "<string 21> <string 22> <...>" ;

translates to the following bag of words grammar:

!grammar bag_of_words;
!start  <sentence_0> <sentence_1>;
<terminal_0> : "<string 11>"!id(0x1) | "<string 12>"!id(0x1) | "<string 13>"!id(0x1) | "<...>"!id(0x1);
<sentence_0> : <terminal_0>+;
<terminal_1> : "<string 21>"!id(0x2) | "<string 22>"!id(0x2) | "<...> "!id(0x2);
<sentence_1> : <terminal_1>+;

Bag Of Words result

The bag of words result is an ordinary NBest result. In case a user ID is given the corresponding recognized item contains the keys “userID” holding the user ID. Otherwise the user ID keys are not contained in the result.

WAKE UP WORD DYNAMIC CONTENT CONSUMER

The wake up word dynamic content consumer implements the wake up word grammar modeling.

Configuration Options

The following shows the configuration for the DCC type wuw:

{
  "version": "VoConHigh 1.0",
  "dynamic_content_consumer": [
    {
      "name": "DCC_WUW",
      "content": [
        {
          "type": "string",
          "name": "wuw_Ortho"
        },
        {
          "type": "integer",
          "name": "wuw_ID"
        }
      ],
      "type": "wuw",
      "wuw": {
        "id": "wuw_ID",
        "lexicon": {
          "uses": "<a lexicon>"
        },
        "context": {
          "acmod": "<an acoustic model>",
          "name": "wuw.fcf"
        }
      }
    }
  ]
}

The key id must link with the content which has _ID postfix and the type of integer, or integer64, or base64.

the DCC adds the new wake-up word to the WUW context or replaces the existing. If the context didn’t exist it will be created. Any existing context will be overwritten.

The wake-up word context is always persistent.

WUW grammar

The WUW context (in “context” object) needs to be created following the BNF rules.

Example of such one-shot WUW grammar:

#BNF+EMV2.1;
!grammar wuw;

!start <wuw>;

<wuw>: !tag(WUW, your wake-up word here) !optional(<...>);

Using COP files as input

Dynamic input data can be provided by COP files. The user ID must be in the COP Entry_Id column.

That field can be a unsigned 32-bit or 64-bit integer: use #NumIDFormat(Entry_ID) Decimal
or a base64 encoded 64-bit integer: use #NumIDFormat(Entry_ID) BSBase64
At this point the NumIDBinLength format specifier is not used. Any bitlength will be accepted
and ultimately user_id_range determines how many bits are stored in the buffer for the user ID.

Anti WUW

There are anti-wake-up words that can also be added to the dynamic WUW context.

These anti-wake-up words are typically words that are phonetically close to the actual wake-up word. Including those anti-wake-up words can help in reducing the false acceptance rate, but it will probably also increase the false rejection rate

Wake-up words and anti-wake-up words can be given to the DCC in the same COP file. A flag at the beginning of each line indicates the words as wake-up word or anti-wake-up word.

Wake-up words use AddEntryMode: a

Anti-Wake-up words use AddEntryMode: r

Example:

  • a;1001;”Hello Auto”;
  • a:1002;”Hello Mercedes”;
  • r:2002;”Hello Merkedes”;

Id’s of anti-wake-up words will not be added to the wake-up word grammar although they can be given in the cop file. The reason is that the recognizer will not give back an result if an anti-wake-up word is recognized.
