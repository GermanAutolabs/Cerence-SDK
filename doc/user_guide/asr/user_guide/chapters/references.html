JSON Configuration reference

The full JSON configuration reference can be found here: Cerence ASR JSON configuration reference. This reference lists all the keys that can occur in a Cerence ASR configuration file, and gives a brief description of what they are used for.

Cloud services

A cloud services object contains the data needed to set up an actual server connection. Typically, cloud services configurations go in te cloudservicesconfig.json file. Note that you can have several cloud services configurations for the same server. This is required if you want to run cloud commands in parallel: commands using the same cloud services object will be executed one after the other, so if you want to run commands in parallel, you need to use different cloud services objects.

An example for HTTP connections:

{
  "cloud_services": {
      "type" : "http",
      "name" : "MyCloudServices",
      "url" : "https://...:443/NmspServlet/",
      "app_id" : "...",
      "app_key" : "...",
      "ssl_certificate_bundle": "bundle.crt",
      "app_name" : "dragondrive_eval_1.0",
      "app_version" : "1.0",
      "app_language" : "eng-USA",
      "carrier" : "n/a",
      "device_model" : "n/a",
      "uid" : "dragon_drive_sdk_tests_uid",
      "input_audio_format" : {
        "encoding" : "ENCODING_OPUS",
        "sample_rate" : 16000,
        "samples_per_channel" : 1920,
        "channel_count" : 1
      },
      "output_audio_format" : {
        "encoding" : "ENCODING_OPUS",
        "sample_rate" : 16000,
        "samples_per_channel" : 1920,
        "channel_count" : 1
      }
    }
}
cloud_services
This is the type of the object. This object could be referenced by its name. This object holds the parameters related to the connection to a server. For details on the configuration see the document nds_cloudservices_configuration_classes.html.
url, app_id, app_key
These are the credentials needed to connect to a cloud server. You will receive these when you order cloud services from Cerence.
ssl_certificate_bundle

This is a pointer to the file that contains the SSL certificates. If this is a relative path, it is resolved relative to the path in which this configuration file resides.

Cloud uploads: structured content

If you want to upload the data with the structured_content type format then you need to specify the key structured_content in the remote DCC configuration. If this key is absent then the old contacts data format is used.

The structured_content configuration maps the content of the COP file to the JSON format that will be uploaded to the server. The server expects the data in the following format:

list: [
        {
          "content_id" : <INT> unique id to identify this item,
          "content" :
          {
          },
        }, ...
     ]

e.g.

[ { "content_id": 12235,
    "content": { "song" : "Thriller", "artist": "Michael Jackson",
                 "album": "MJ greatest hits", "lastPlayed": "2012-08-24"}
  },
  { "content_id": 12452,
    "content": { "song" : "abc", "artist": "Michael Jackson",
                 "album": "MJ greatest hits", "lastPlayed": "2012-09-01"} }
  },
  ... ]

or

[ { "content_id": 525,
    "content": { "street": "Maisonneuve", "ShopName": "Starbucks", "opened24h": 1 }
  },
  { "content_id": 526,
    "content": { "street": "Sherbook", "ShopName": "Cafe Express", "opened24h":0 }
  },
  ... ]

The actual key names ("song", "street", …) in the "content" object depend on the command sent to on the server. The structured_content key in the DCC configuration maps these names to the names in the COP file.

An example will make this clear. Below is a DCC configuration for first and last names. In the COP file, the columns have names in all caps while the server command expects key names in all lower case.

{
  "dynamic_content_consumer": {
    "name": "CONTACTS",
    "content": [{
      "name": "Contact_ID",
      "type": "integer"
    },{
      "name": "FirstName_Ortho",
      "type": "string"
    },{
      "name": "LastName_Ortho",
      "type": "string"
    },{
      "name": "Company_Ortho",
      "type": "string"
    },{
      "name": "PhoneTypes_Ortho",
      "type": "string_list"
    }],
    "type": "remote",
    "remote": {
      "cloud_services_name": "csHttp",
      "command": "DRAGON_NLU_DATA_UPLOAD_CMD",
      "grammar_type": "structured_content",
      "structured_content": {
        "category" : "contacts",
        "id": "Contact_ID",
        "content" : [{
          "name": "fn",
          "link": {
            "name": "FirstName_Ortho"
          }
        },{
          "name": "ln",
          "link": {
            "name": "LastName_Ortho"
          }
        },{
          "name": "c",
          "link": {
            "name": "Company_Ortho"
          }
        },{
          "name": "ph",
          "link": {
            "name": "PhoneTypes_Ortho"
          }
       }]
      }
    }
  }
}

The "content" array in the "structured_content" config object links the name as expected by the server (fn), to the column name in the COP file (FirstName_Ortho).

NOTE: For the NCS_UPLOAD_DATA_CMD command, the configuration is slightly different. In this case, it is important to specify the mode. Additionally, the "grammar_type" key will get the value that is specified in the "category" key in the example above, and "category" key should be removed or empty.

File formats

COP files

The COP file definition can be found here FileFormat_COP.pdf.

HRL files

The ‘hash reference list’ (hrl) files describe a list of utterances, the speakers of those utterances, the gender of those speakers and what the speakers have said. The files can contain more information; for example, whether the reference word sequence has been validated, what the start and end times of the words in the reference word sequence are, etc.

For that purpose, you should create a file with extension .hrl that has the syntax that is described below.

Here is a fragment of such a file:

#head;hrl;2.0;utf-16
#ref#speechfile#speaker#gender#reference word sequence
head
ref#sound1.wav#spk1#male#three six five two
ref#sound2.wav#spk2#male#four two one eight
ref#sound3.wav#spk3#female#one five seven
ref#sound4.wav#spk4#female#two five four

The first line of the file always starts with #head;hrl;<version>;<encoding>. The current version number is 2.0. Different encodings are supported.

We recommend to use an Unicode encoding for your hrl files, and, if you create the hrl file in an editor, use an editor that includes a correct Unicode mark in the beginning of the file.

Note that if the encoding is utf-16, the file must start with a byte order mark. This byte order marks allows the tools to figure out whether the file uses little-endian or big-endian byte order.

The second line specifies the format of the reference lines. Each reference line starts with ref and contains information that is specific to one particular speech utterance. In this example each reference has 4 fields.

“speechfile” - The relative path to a speech file.

The Batch Tool will derive the absolute path to a speech file by concatenating a value of “speechfile” to the value of “sound_dir”.

The supported audio file formats are listed below and configured by including their extentions:

  • pcm: raw pcm, linear, mono, 16000 Hz, 16 bit Intel (LSB first in the file, also known as ‘little endian’)
  • pcmswap: raw pcm, linear, mono, 16000 Hz, 16 bit Motorola (MSB first in the file, also known as ‘big endian’)
  • wav: Microsoft wave: Windows PCM, mono, 16000 Hz, 16 bit Intel
  • nist: soundfiles with an NIST header. At least the following format will work: linear pcm, mono, 16000 Hz, 16 bit Intel. “Shorten” decompression has been disabled.

“speaker” - Name of the speaker. This may just as well be any alfa-numeric speaker identification

“gender” - the gender of the speaker. Following genders are available: - unknown - male - female - child (i.e. boy or girl) - boy - girl

“reference word sequence” - A sentence describing what the speaker has said in that utterance.

A number of special characters must be escaped when they are used in value fields and should be interpreted as literal characters instead of serving a special purpose. The field separator #, as well as the escape character itself, must be escaped using the escape symbol in all value fields. Also the carriage return and new-line must be escaped as r and n respectively when used in value fields. Also the symbol ; must be escaped because it serves as second level separator ; within fields that represent a list of items of the same type.

The speechfile, speaker and reference word sequence fields are the only mandatory fields for an hrl that can serve as input for the Batch Tool.

Match file

A ‘match’ (.res) file contains information about a recognition result. It is produced by the batch tool, and used as input to a match file parser.

Formatting elements

The file consists of a header specifying the different record types, and a body containing instances of these record types, which must conform to the definition in the header. Header records start with a ‘#’, followed by the record name, and all field names, separated by ‘#’. Each ‘#’ inside field value (i.e. non-separator) will be replaced by ‘#’, so that it is not treated as a separator. Field names starting with a ‘;’ indicate that the field is a ‘;’-separated array, rather than a single value.

A number of special characters must be escaped when they are used in value fields and should be interpreted as literal characters instead of serving a special purpose. The field separator ‘#’, as well as the escape character ‘’ itself, must be escaped using the escape ‘’ symbol in all value fields. Also the carriage return and new-line must be escaped as r and n respectively when used in value fields. Also the symbol ‘;’ must be escaped because it serves as second level separator ‘;’ within fields that represent a list of items of the same type.

Header

The file starts with a header describing the further contents. Here is the full ‘match’ header:

#head;res;1.0;utf-8#soundFiledir#hrl filepath
#hyp#rank#;hypothesis#confidence
#res#json
#ref#speechfile#speaker#reference word sequence#number of hypotheses
head#unspecified#unspecified

The first line of the file always starts with #head;res;<version>;<encoding>. The current version number is 1.0. Different encodings are supported.

We recommend to use an Unicode encoding for your hrl files, and, if you create the hrl file in an editor, use an editor that includes a correct Unicode mark in the beginning of the file.

Note that if the encoding is utf-16, the file must start with a byte order mark. This byte order marks allows the tools to figure out whether the file uses little-endian or big-endian byte order.

The last line of the example header is the ‘head’ instance. Each match can only have one such head instance, which must immediately follow the definition part of the header.

Reference (‘ref’) records

Reference (‘ref’) records have the following definition:

#ref#speechfile#speaker#reference word sequence#number of hypotheses#<extra columns>

An example ref record instance is:

ref#call_nancy_brooks.pcm#spk1#call nancy brooks#3#male

“speechfile” - The relative path to a speech file.

The Batch Tool will derive the absolute path to a speech file by concatenating a value of “speechfile” to the value of “sound_dir”.

The supported audio file formats are listed below and configured by including their extentions:

  • pcm: raw pcm, linear, mono, 16000 Hz, 16 bit Intel (LSB first in the file, also known as ‘little endian’)
  • pcmswap: raw pcm, linear, mono, 16000 Hz, 16 bit Motorola (MSB first in the file, also known as ‘big endian’)
  • wav: Microsoft wave: Windows PCM, mono, 16000 Hz, 16 bit Intel
  • nist: soundfiles with an NIST header. At least the following format will work: linear pcm, mono, 16000 Hz, 16 bit Intel. “Shorten” decompression has been disabled.

“speaker” - Name of the speaker. This may just as well be any alfa-numeric speaker identification.

“reference word sequence” - A sentence describing what the speaker has said in that utterance.

“number of hypotheses” - The number of hypotheses of an ASR result.

“<extra columns>” - put the other columns from the reference line from hrl file here. for example:

#ref#speechfile#speaker#reference word sequence#number of hypotheses#gender

ASR Hypothesis (‘hyp’) records

ASR hypothesis (‘hyp’) records have the following definition:

#hyp#rank#;hypothesis#confidence

An example hyp record instance is:

hyp#0#dial;mom;cell#5075

A number of ‘hyp’ records following one ‘ref’ record forms an (ASR) n-best list.

“rank” - The rank number in the ASR n-best list.

“;hypothesis” - A ‘;’-separated list of tokens forming the ASR hypothesis.

“confidence” - The ASR confidence for this hypothesis.

Res (‘res’) json records

Res json records have recognition json results, including ASR and NLU results, as well as the latency json.
