Recognition Results

Result types

Result events

The recognizer communicates its recognition results through a callback function, onResult, that is one member of the IRecognizerListener structure. Because the onResult callback is tied to the recognizer, it will handle all the results that any of the ASR applications used with that recognizer can return.

All results are in JSON format, but depending on the type of ASR applications used with a recognizer, the structure of a result will be different. There are ASR results, NLU results, and Cloud results. ASR results can also be partial (intermediate), if Cerence ASR is configured to also provide partial results.

Consequently, your onResult implementation should be capable of handling all the different types of results that you expect your ASR applications to return. In the following sections, the structure of each result type will be explained.

ASR results

An example result from an ASR recognition would look like this:

type: 'RESULT_TYPE_ASR',
result: {
    "status": "stable",
    "type": "NBest",
    "version": "1.0.0",
    "beginTime": 0,
    "endTime": 1940,
    "activeStartRules": ["SLM#START"],
    "hypotheses": [{
            "startRule": "SLM#START",
            "beginTime": 420,
            "confidence": 7133,
            "endTime": 1940,
            "lmScore": 2944,
            "score": 77454,
            "items": [{
                    "orthography": "call",
                    "type": "terminal",
                    "beginTime": 440,
                    "confidence": 6498,
                    "endTime": 770,
                    "lmScore": 1149,
                    "score": 17030
                }, {
                    "orthography": "nancy",
                    "type": "terminal",
                    "beginTime": 770,
                    "confidence": 7594,
                    "endTime": 1220,
                    "lmScore": 721,
                    "score": 22970
                }
            ]
    }]
}

This is an NBest result that contains a single hypothesis. At the top level, the result contains the start and the end timestamps of the audio for which this result contains hypotheses (beginTime and endTime). Also contained at the top level is the list of start rules, from all search contexts, that were active at the time the result was produced.

Each hypothesis contains the context start rule that was actually used in producing this result. In the example, the startRule is SLM#START. This rule allows you, if you have a search space composed of several contexts (e.g. if you have several ASR applications active in parallel), to determine which context produced the hypothesis. In SLM#START, the context is called SLM, and START is the start rule in that context.

A hypothesis also contains start and end timestamps, that further narrow the audio window to precisely the terminals in the hypothesis. The hypothesis also has an overall confidence that is calculated from the confidences of the individual terminals. The confidence is what you typically use to judge a hypothesis. In contrast to the other measures included, such as score and lmscore, the confidence is normalized across hypotheses, and can be used to compare several hypotheses against each other.

Finally, a hypothesis contains an array of items. Items can be terminals ("type": "terminal"), but they can also be tags. Terminal items have an orthography, which is transcribed from a section of the audio stream. Tags will contain nested items, such as other tags or terminals. The items thus define a tree-like structure of items recognized in the audio stream, or matched in the context (tags do not appear in the audio stream, but are always composed of other items that have been recognized in the audio stream).

NLU results

If you configure your ASR application with an nlu section, then every time a final ASR result is created, an NLU result is created as well. Your onResult callback will be called two times, first with the ASR result, and then with the corresponding NLU result. This works this way because an NLU result is always based on an ASR result that precedes it.

This is what the NLU JSON result looks like:

{
  "type": "SemanticResult",
  "versions": {
    "SemanticRepresentation": {
      "version": "0.1.1"
    },
    "SemanticResult": {
      "version": "3.0.0"
    },
    "SemanticSpecification": {
      "version": "1.0.0"
    },
    "VoCon": {
      "version": "5.1"
    }
  },
  "results": [{
    "appName": "",
    "probability": 10000,
    "interpretations": [{
      "id": "b7512171-ac0a-4e6a-b8a6-17583b807ea6",
      "probability": 10000,
      "representation": {
        "name": "__ITN__",
        "type": "concept",
        "extraContent": {
          "VoCon": {
            "slotName": "topic"
          }
        },
        "roles": {
          "FreeText": {
            "type": "constant",
            "value": "123",
            "extraContent": {
              "VoCon": {
                "slotName": "FreeText"
              }
            },
            "terminalIndices": [0, 1, 2]
          }
        }
       }
    }]
  }],
  "terminals": [
    {
      "orthography": "one\\number",
      "beginTime": 1450,
      "confidence": 6051,
      "endTime": 2050,
      "score": 40086
    },
    {
      "orthography": "two",
      "beginTime": 2150,
      "confidence": 7205,
      "endTime": 2610,
      "score": 29246
    },
    {
      "orthography": "three",
      "beginTime": 3060,
      "confidence": 7233,
      "endTime": 3660,
      "score": 38602
    }
  ]
}

The type will be SemanticResult. For ASR results, this is NBest. Note that the onResult callback in the IRecognizerListener has an argument called resultType of type nuance_asr5_ResultType which indicates the result type, so you don’t need to extract the type field in order to check for the result type.

At the end of the JSON structure, there is a section called terminals. This section contains an array with all the terminals that appear in the ASR result on which this NLU result is based. The semantic interpretations refer to the terminals by index in this array, rather than repeating the full terminal structures. (See terminalIndices in the section roles of the JSON structure.)

The section result will have the actual semantic interpretation(s) of the ASR result, in the interpretations array. Each JSON block in this array is an alternative interpretation of the ASR result.

  "status_code": 0,
  "result_format": "rec_text_results"
}

The key final_response indicates that this is a final response.

The key words has the detailed recognition results, as a JSON array of JSON arrays. Each “innermost” array has the list of words for that interpretation. The transcriptions array has the transcription of the interpretations, in the same order as the words array. The confidences array again has the confidences for each interpretation in the words and transcriptions arrays.

The value for the key words is an array of arrays. Each “inner” array is a hypothesis; it contains a JSON block for each word that lists the confidence and the recognized word:

[{
  "confidence": "0.999",
  "word": "Hello\\*no-space-before"
},{
  "confidence": "0.977",
  "word": "Dragon"
}]

(Note the \\* which precedes “no-space-before”. The word transcriptions can contain directives, for example for use by TTS systems.)

The value for the transcriptions key contains an array of strings, which are the textual representations of the hypotheses listed in the words key.

Partial vs. final results

(For partial results from the cloud, see here.)

Cerence ASR is designed to run as a stream-based recognizer. ASR used to be used for relatively short data entry sessions, such as command entry after a button push (“push-to-talk”) or after detecting a wake-up-word. For such short entries, and when there is a relatively well-defined set of possible phrases to be recognized, it makes sense to wait for the end of the utterance, and produce a single recognition result at the end of the input.

However, use cases for voice data entry are expanding, and also on embedded platforms, voice recognition is more and more often used for tasks such as message dictation, or one-shot entry of complex input such as a route navigation destination. In these cases, the input may be very long, and it becomes important for the system to be able to show progress to the user. This serves several purposes:

  • it is an indication for the user that the system is actually still listening. Long periods of inactivity may cause the user to doubt whether the system is still listening.
  • if the system can show the user what it has recognized up to then, this allows the user to review what they have said up that point.
  • additionally it gives an indication to the user of the accuracy of the recognized speech, which gives the user the opportunity to evaluate if it makes sense to continue speaking.

Cerence ASR enables these scenarios by optionally providing partial recognition results to the system.

Configuring the engine for partial results

By default, Cerence ASR will only return final results. If you want to receive partial results, you have to configure Cerence ASR to do by setting the parameter LH_SEARCH_PARAM_STREAM_RESULT_MODE to 1 or 2. This parameter should be set in the search configuration of an ASR application, as follows:

"application" : [{
  "name" : "APP_CNC",
  "input": [{
    "type" : "asr",
    "asr" : {
      "search" : [{
          "name" : "CNC_CTX",
          "type" : "static",
          "static" : {
            "file_name" : "modelm_small_fmt2.fcf"
          },
          "vocon_parameters": {
             "LH_SEARCH_PARAM_STREAM_RESULT_MODE": 1
          }
        }
      ]
    }
  }]
}]

The allowed values of LH_SEARCH_PARAM_STREAM_RESULT_MODE are these:

  • 0 (LH_SEARCH_STREAM_RESULT_DISABLED)
  • 1 (LH_SEARCH_STREAM_RESULT_INTERMEDIATE)
  • 2 (LH_SEARCH_STREAM_RESULT_STRIPPED_INTERMEDIATE)

When using LH_SEARCH_STREAM_RESULT_INTERMEDIATE, the intermediate results will contain both stable words and non-stable words. Stable words are words that the recognizer will not change any more as recognition continues. Non-stable words are included in an intermediate result as a “current best guess” hypothesis, and they may change as recognition continues and the recognizer finds better results for those words.

If you don’t want to have the stable part of the recognition result to be repeated in each intermediate result, you can use the LH_SEARCH_STREAM_RESULT_STRIPPED_INTERMEDIATE result mode. In this mode, only new stable words and non-stable words will be included in the results. Previously stable parts of the result are not repeated.

Partial result contents

First of all, if a result is non-final, this will be indicated in the isFinal parameter of the onResult callback. This is the API definition of the onResult callback in the IRecognizerListener interface:

typedef void(*FUNC_nuance_asr5_IRecognizerListener_onResult) (void *instance, const LH_CHAR *result, nuance_asr5_ResultType resultType, LH_BOOL isFinal)

When this callback function is called by the recognizer, the final argument, isFinal indicates whether the result is a final result or not.

This is an example of a partial result (note that all non-relevant key-value pairs have been removed for clarity):

{
  "status": "unstable",
  "hypotheses": [{
    "startRule": "SLM#START",
    "confidence": 5330,
    "items": [
      {
        "orthography": "one\\number",
        "confidence": 6051,
        "stability": 10000
      },
      {
        "orthography": "two",
        "confidence": 7205,
        "stability": 5000
      },
      {
        "orthography": "three",
        "confidence": 7233,
        "stability": 5000
      }
    ]
  }]
}

The result has a status keyword: if it has the value unstable, the result is partial, and has terminals that are not stable. For a final result, status will have the value stable.

In the NBest result, every terminal will have, next to a confidence value, also a stability value that indicates whether the engine may still replace the terminal with a different one. Stabilized terminals will have a stability value of 10000; if the value is lower, the terminal is not stable and may be replaced in subsequent partial or final results.

So, in the example above, one is stable, and two and three are not (yet) stable, even if they have high confidence values.

The rule of thumb when using partial results, is that the final utterance is the sequence of terminals that have, in any of the partial results, reached a stable state.
