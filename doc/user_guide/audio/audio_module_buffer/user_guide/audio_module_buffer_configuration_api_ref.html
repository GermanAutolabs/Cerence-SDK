Overview

The following is an overview of the structure of the JSON configuration files.

  • { ITopLevelAudioConfig }
    • version: String
    • label: String
    • audio_scenario: [ { IAudioScenarioConfig } ]
      • name: String
      • audiopaths: [ [ String ] ]
    • audio_plugin: [ { IAudioPluginConfig } ]
      • type: String
      • lib_name: String
    • module_config: [ { IAudioModuleConfig } ]
      • name: String
      • { IAudioBufferConfig }
        • buffer_time: int
    • audio_format: [ { IAudioFormatConfig } ]
      • name: String
      • sample_rate: int
      • samples_per_channel: int
      • channel_count: int
      • encoding: Encoding
      • container: Container

IAudioBufferConfig - Interface of the ‘AudioBuffer’ audio module configuration.

Super type: IAudioModuleConfig

Interface of the ‘AudioBuffer’ audio module configuration.

The IAudioBufferConfig interface represents the configuration of the AudioBuffer audio module.

The AudioBuffer module ensures a minimal size of buffers at a defined position in the audio backbone.

Key Name Value Type Required? Documentation
"name":
String
yes
The name of the audio module.
This property is inherited from IAudioModuleConfig.
"buffer_time": int no, default value is 250 The minimal time in ms which has to be covered by the buffers at this position in the audio backbone

Details:

"name":

The name of the audio module.

This name has to be unique. For instance it is used to reference this module from audio scenarios (audio paths).

"buffer_time":

The minimal time in ms which has to be covered by the buffers at this position in the audio backbone

Expected: the value should cover at least two packages. There is an upper limit of 2min (120000ms) and a warning if configuring more than 10s (10000ms).

Minimum Value: 100 Maximum Value: 120000 Default Value: 250

IAudioScenarioConfig - The audio scenario configuration.

Super type: ITopLevelConfig

The audio scenario configuration.

The IAudioScenarioConfig interface is used to represent the configuration of an audio scenario, i.e. the definition of a set of audio paths (source-sink connections) that can be activated together using the audio manager.

An audio scenario configuration needs to define two items.

  • First, a unique name. This name needs to be used to activate the audio scenario during runtime via IAudioManager.activateScenario(name).
  • Second, an audio path definition, which is a list of source sink connections defined in an array. For such a connection a list with two elements is used. The first element is the name of an audio source, the second element the name of an audio sink.

The array of source-sink lists is building the “audiopaths” configuration parameter. Imagine there is a producer named “mod1”, a processor named “mod2” and a consumer named “mod3”. The definition of the audio path connecting those three modules would look as follows:

"audiopaths": [ ["mod1", "mod2"], ["mod2", "mod3"] ]

The names in that list must match the ones used in the application program when creating a modules instance or refer to a built-in module implementation. Let us assume, for example, that an application creates a prompter instance with the name myprompter:

nuance.prompter.IPrompter.create("myprompter", config, errorListener)

Then this same name myprompter needs to be used in the audio JSON configuration file to define the audio path, for example:

"audiopaths": [ ["myprompter", "default_speaker"] ]

This example defines an audio path where the myprompter source module produces audio data that shall be passed to the consumer module default_speaker which is here supposed to refer to the built-in AudioOutput implementation. This built-in module will be started automatically by the audio manager with the name "default_speaker" if configured as shown and not already present.

Each audio scenario must define at least one source sink connection, using valid module names. Given these audio path definitions the audio manager will set up the required connections for the audio data flow between the modules. Thus, the possibility to define different audio scenarios allows a very flexible configuration of audio processing chains within the audio manager.

Only configured audio scenarios can be activated at the audio manager’s interface. Handing an unknown scenario name will lead to an error.

Key Name Value Type Required? Documentation
"name": String yes The name of the audio scenario.
"audiopaths": [ [ String ] ] yes The audio paths of this audio scenario.

Details:

"name":

The name of the audio scenario.

This name is used for referencing this audio scenario when activating it through the audio manager API.

"audiopaths":

The audio paths of this audio scenario.

An audio path is defined as a list of 2-element lists.

The following naming scheme is used to define a connection in that 2-element lists.

Each sink or source pad is referenced by the name of its module, then followed by a colon (“:”) and the name of the sink or source pad itself. The syntax is module_name:source_pad_name respectively module_name:sink_pad_name.

Note: The colon delimiter therefore cannot be used as part of any module name, sink pad name or source pad name.

For modules having just one sink and/or source pad the name of the sink or source pad does not have to be defined, since the modules name is already a unique key then.

For more complex modules with multiple sink and source pads the audio path configuration needs to use the colon delimiter syntax module_name:source_pad_name respectively module_name:sink_source_name.

IAudioPluginConfig - The audio plug-in configuration.

Super type: ITopLevelConfig

The audio plug-in configuration.

The IAudioPluginConfig interface is used to represent the loadable plugins.

This is important if the built-in “Encoder” module should use a customer encoder implementation, for instance. Since a plugin is only ready to be used in the modules configuration, if it is one of the “known plugin types” listed below and configured in the plugins section.

Adding configurations for other plugin types than listed below will be ignored.

  • Encoder
Key Name Value Type Required? Documentation
"type": String yes type of the plug-in which shall be loaded.
"lib_name": String yes library file name which shall be loaded.

Details:

"type":

type of the plug-in which shall be loaded.

This name is used to define the type of the plug-in this entry is for.

"lib_name":

library file name which shall be loaded.

This name is used for referencing the library which contains the dedicated plug-in implementation.

A plug-in library name may contain an absolute (i.e. starting with / or [A-Za-z]:[/]), a relative or none path information. In the latter two cases the platform specific search strategy (e.g. LD_LIBRARY_PATH on Linux) is used. In case a library name does not contain a platform specific shared object precedence (e.g. lib) or extension (e.g. .so), it is automatically added.

IAudioModuleConfig - The audio module configuration.

Super type: ITopLevelConfig

The audio module configuration.

The IAudioModuleConfig interface is used to represent the generic configuration of audio modules. The derived audio module (configurations) inherit the name property defined here.

It is possible to add dedicated module configurations to the audio manager’s configuration file.

This is important if the built-in “AudioInput”/”AudioOutput” should be used, for instance. Since a module is only ready for automated internal startup, if it is one of the “known modules” listed below and configured in the modules section.

Adding configurations for other module types than the “known modules” listed below will cause an error while loading this configuration.

  • AudioBuffer
  • AudioDecoder
  • AudioEncoder
  • AudioFromFile
  • AudioInput
  • AudioOutput
  • PromptSyncedInput (FW only)
  • AudioSegmentation
  • AudioSignalAnalysis (FW only)
  • SampleRateConverter
  • SpeechSignalEnhancement
  • AudioStreamDistributor
  • AudioToFile

Currently known module types are listed as “sub types”, see the documentation of the sub-types for the list of type-specific properties.

Note: Configuring modules of those types and referencing their names in the audio path of a scenario will allow the implicit creation of a module instance if no module with that name is present so far. The implicit creation is done during the preparation step of an audio scenario activation. Please choose the names for your modules carefully to avoid name conflicts.

Key Name Value Type Required? Documentation
"name": String yes The name of the audio module.

Details:

"name":

The name of the audio module.

This name has to be unique. For instance it is used to reference this module from audio scenarios (audio paths).

IChunkPoolConfig - ChunkPool configuration.

ChunkPool configuration.

Key Name Value Type Required? Documentation
"chunk_count": int no, default value is 100 The initial number of chunks per audio format that are allocated.
"chunk_realloc_count": int no, default value is 50 The number of audio chunks used for reallocations.
"chunk_threshold": int no, default value is 2024 Threshold for dynamic versus static allocation within an audio chunk.
"marker_count": int no, default value is 100 The initial number of audio markers that gets allocated.
"marker_realloc_count": int no, default value is 100 The the number of audio markers used for reallocations.

Details:

"chunk_count":

The initial number of chunks per audio format that are allocated.

This number of audio chunks of a certain audio format gets allocated the first time a reserve is made for this audio format. The number of allocated chunks will never fall below this value, even if all reservations are cleared.

"chunk_realloc_count":

The number of audio chunks used for reallocations.

This number of audio chunks of a certain audio format gets allocated when the number of reservations exceeds the current number of allocations. The same number is also used for decreasing the number of allocated chunks when reservations are cleared.

"chunk_threshold":

Threshold for dynamic versus static allocation within an audio chunk.

This value only takes effect for chunks carrying encoded audio data. If the chunk size is below the threshold it is tried to re-use already allocated memory while bigger audio chunks are allocated dynamically. For PCM audio data a complete static approach is used.

"marker_count":

The initial number of audio markers that gets allocated.

This number of audio markers gets allocated the first time a marker is requested.

"marker_realloc_count":

The the number of audio markers used for reallocations.

This number of audio markers gets allocated when an audio marker is requested and no more unused marker is available.

IAudioFormatConfig - The audio format configuration.

The audio format configuration.

This configuration can be referenced with ("uses") from other parts of the configuration using the name key. The name of audio format has to be unique.

Key Name Value Type Required? Documentation
"name": String no The name of the audio format.
"sample_rate": int yes The sampling rate in Hz.
"samples_per_channel": int yes The number of units per channel.
"channel_count": int yes The number of channels.
"encoding": " Encoding " no, default value is ENCODING_PCM_16BIT The encoding type.
"container": " Container " no, default value is CONTAINER_NONE The container type.

Details:

"name":

The name of the audio format.

This key is used for referencing this audio format from other parts of the configuration.

"sample_rate":
The sampling rate in Hz.
"samples_per_channel":

The number of units per channel.

This key defines the samples per channel for plain PCM data (ENCODING_PCM_16BIT) without a container (CONTAINER_NONE). In all other cases it defines bytes per channel.

"channel_count":
The number of channels.
"encoding":

The encoding type.

" Encoding ":

  • "ENCODING_USER": user specific encoding type, unknown by the system
  • "ENCODING_PCM_16BIT": Signed 16Bit PCM raw data
  • "ENCODING_OPUS": OPUS encoding
  • "ENCODING_MP3": MP3 encoding
"container":

The container type.

" Container ":

  • "CONTAINER_USER": user specific container type, unknown by the system
  • "CONTAINER_NONE": No container used/desired
  • "CONTAINER_OGG": OGG container
