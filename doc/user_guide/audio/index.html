Scope of Document

The main scope of this documentation is to serve as a developer guide for Cerence Professional Services (PS) engineers and for customers who want to use the audio framework of the Cerence SDK.

Note

API call examples in this document are shown in C syntax.

Overview

The audio framework provides access to Cerence’s state-of-the-art audio processing components through high-level easy-to-use application programming interfaces (API) that enable fast-track development of complex audio use cases.

The audio framework concept is simply based on a set of audio modules and a central Audio Manager component. Thereby each audio module is providing a specific audio processing capability like for instance audio i/o, audio coding, speech signal enhancement, sample rate conversion, speech recognition, text-to-speech synthesis or such the like. The Audio Manager on the other hand provides among others the possibility to connect audio modules to specific audio processing chains to realize different audio use cases. These use case specific audio module chains can be predefined in a JSON configuration file of the Audio Manager in so called audio scenario definitions. The Audio Manager provides a corresponding API to connect the audio modules predefined in these audio scenario definitions.

In addition the Audio Manager acts as a central registry of audio adapters, audio codecs and audio containers that can be used from specific audio modules to perform external audio i/o and decoding, encoding, packing and unpacking of audio data.

The audio framework components are configurable via JSON configuration file(s) to the needs of the audio use cases they are used for.

Basic Concepts

External Audio Integration

The audio framework uses an audio adapter approach to realize the access to external audio sinks and sources (e.g. audio devices). These audio adapters can be integrated as subcomponents of the Audio Input and Audio Output audio modules in order to stream audio into or from an audio processing chain defined in an audio scenario. Thereby the binding of a specific audio adapter to a dedicated Audio Input or Audio Output module instance is configurable via the corresponding module JSON configuration by setting the unique adapter type name. For more details see chapter Audio Adapters.

Note

As audio adapter implementations are highly platform and project specific the SDK delivery does contain only sample audio adapter implementations for each standard platform SDK package. These sample adapters are provided as source code and may serve as a starting point for Cerence PS or customer engineers to develop project specific adapter(s). Note that the sample adapters are not guaranteed to work on each specific platform. Finally it is the responsibility of Cerence PS or the customer to implement and test a properly working external audio integration in the form of an audio adapter.

These external audio implementations must fulfill the CSDK Audio Requirements.



Audio Codecs and Containers

The SDK is delivered with a set of audio codecs (e.g. Opus) and containers (e.g. Ogg). In case these codecs and containers do not meet the project requirements the audio framework provides a possibility to implement and add other codecs respectively containers.

In general all these audio codecs and containers can be integrated as subcomponent of the Audio Decoder and Audio Encoder audio modules into an audio processing chain defined in an audio scenario. Thereby the binding of a specific audio codec and container to a dedicated Audio Decoder or Audio Encoder audio module instance is configurable via the corresponding module JSON configuration by setting the unique type name of the codec or container. For more details see chapter Audio Codecs and Audio Containers.



Audio Module Creation and Destruction

In general there are two different methods to create and destroy audio modules. One is the so called implicit method that is the propagated way and the other is the explicit method that should be used only if necessary. Which method is supported by which module is described in chapter Audio Modules.

The implicit creation method is a feature of the Audio Manager that automatically creates audio module instances during the activation of an audio scenario. As a precondition for using implicit module creation the application needs to register the corresponding factory at the Audio Manager. For that purpose all audio modules which support implicit creation provide a registerFactory API function. For details see chapter Registering Audio Module Factories.

An audio module instance is created implicitly by the Audio Manager in context of an audio scenario activation when the following conditions are met:

  • the corresponding audio module factory is registered at the Audio Manager
  • the audio module is part of the audio scenario being activated
  • an audio module instance, having this instance name, does not already exist

Implicitly created audio module instances are automatically destroyed on the deactivation of the audio scenario they are part of.

The explicit creation is the way that the application instantiates and destroys an audio module instance via its specific API. This has to happen before the instance is used in an audio scenario context. Explicitly created module instances need to be destroyed by the application after the audio scenario they are used in is deactivated.

Note

Both ways to create audio modules can be used in a mixed way. This means, an application can create some audio modules explicitly, whereas others are created implicitly during the activation of an audio scenario. As a rule of thumb the application developer generally should use the implicit method. The explicit method should only be used in case the application needs to control the specific audio module via its specific API. For instance the Asr speech recognizer always needs to be created explicitly because the application has to control the module via its API.



Audio Module Connection

All audio modules can be chained in a flexible way to realize specific audio processing use cases. The definition of such audio chains can be done in so called audio scenario definitions in a JSON configuration file of the Audio Manager. Each audio scenario definition must have a unique name that can be used to activate and deactivate it at the Audio Manager. During an activation of an audio scenario all involved audio modules are connected automatically by the Audio Manager, whereas on deactivation they become disconnected again. For more details see chapter Connecting Audio Modules.

The following survey schematically shows the required steps to define and use an audio scenario in order to connect audio modules to a use case specific audio processing chain:

  • The specific audio scenario that shall cover an audio use case needs to be defined in a JSON configuration that is used to configure the Audio Manager (1).
  • Whenever this audio scenario is needed the application can activate it (2) at the Audio Manager API. By the same token an activated audio scenario can be deactivated when it is no longer needed (4).
  • In context of an activation the Audio Manager connects the involved audio module instances defined in the specific audio scenario (3). Audio module instances and related subcomponents (e.g. adapters, codecs) that do not already exist are created implicitly when a corresponding factory was registered at the Audio Manager beforehand.
  • On deactivation of a scenario the involved audio module instances are disconnected(5). Audio module instances and related subcomponents (e.g. adapters, codecs) that were previously created implicitly on activation are destroyed automatically.


Note:

The audio framework of the SDK is designed to operate for speech dialog systems only.

This means that:

  • it is designed to hand audio data from an external real time audio device to an engine like the ASR, operating faster than real time.
  • it is designed to hand audio data from an engine like the TTS, operating faster than real time, to an external real time audio device.
  • it is NOT designed to hand audio data from an external real time audio device to an external real time audio device. The audio framework works asynchronous, therefore it is not guaranteed that one packet of audio data handed in, from an external audio device, via the audio input, triggers immediately the emission of a corresponding packet of audio data, via the audio output, to an external audio device.

Example:

The following shall illustrate an overall example from definition of a specific audio scenario in the JSON configuration to using the programmatical API to activate and deactivate the defined audio module chain.

First of all an corresponding audio scenario needs to be defined in the audio JSON configuration. Such a scenario definition contains a name parameter that is used to activate and deactivate the specific audio scenario and an audiopaths parameter that allows to define a list of audio module instances that shall be connected to each other. Here, the audio scenario named recognitionScenario shall connect an Audio Input module instance microphone with an Asr module instance recognizer to realize a simple recognition audio processing chain. In addition the Audio Input module instance microphone shall use an instance of an audio adapter implementation of type asio_audio_adapter that realizes an external audio integration as already described in chapter External Audio Integration. This is configured in the module configuration of the Audio Input module instance microphone with parameter adapter_type. The configuration of the Asr module instance recognizer is not shown in this JSON example configuration.

Audio Manager

Description

The Audio Manager is the central audio management component with the following functions:

  • setting up general audio framework infrastructure needed by all Audio Modules
  • providing a central registry for:
    • Audio Modules
    • Audio Adapters
    • Audio Codecs
    • Audio Containers
  • providing an API to connect and disconnect Audio Modules
  • implicit Audio Module creation and destruction

Developing Applications

API Reference

The API reference documentation of the SDK Audio Manager, which is part of the audio core component, is included in the Cerence SDK package at doc/api_reference/audio/audio_core/. There is a separate folder for each supported programming language in which the reference documentation can be accessed via the main page index.html.

The C API documentation is available at doc/api_reference/audio/audio_core/c/index.html.

Configuration API Reference

The configuration has to be done in the JSON format as specified in the configuration API reference:

  • Overview
  • ITopLevelAudioConfig - The top-level audio configuration structure.
  • ITopLevelConfig - Interface for top-level configuration classes.
  • IUserJsonConfig - Defines a verbatim block of json that is used by VoCon High as is.
  • IAudioScenarioConfig - The audio scenario configuration.
  • IAudioPluginConfig - The audio plug-in configuration.
  • IAudioModuleConfig - The audio module configuration.
  • IChunkPoolConfig - ChunkPool configuration.
  • IAudioFormatConfig - The audio format configuration.

Creating and Configuring

An Audio Manager instance is created with the function nuance_audio_IAudioManager_create(). Before calling the nuance_audio_IAudioManager_create() function a nuance_common_IConfiguration instance containing the Audio Manager’s JSON configuration needs to be created. This is done by loading the JSON configuration as described in Common Configuration Loading. Additionally the Audio Manager needs a System Manager handle. For details see System Manager Creation.

As central component of the audio framework the Audio Manager has to be created at the first place before any audio module.

Connecting Audio Modules

The Audio Manager supports the activation and deactivation of so called audio scenarios to connect audio modules to audio processing chains that realize specific audio use cases. Audio scenarios are defined in the JSON configuration file of the Audio Manager. An audio scenario definition contains a name parameter that is used to activate and deactivate this specific audio scenario. The audiopaths parameter allows to define a list of audio paths. Whereas each audio path defines a list of audio modules that shall be connected to each other.

Examples of audio path definitions:

  • an audio scenario with 1 audio path connecting the 2 audio module instances a and b:

    [[inst_module_a, inst_module_b]]

  • an audio scenario with 1 audio path connecting the 3 audio module instances a, b and c:

    [[inst_module_a, inst_module_b], [inst_module_b, inst_module_c]]

  • an audio scenario with 2 audio paths connecting the 3 audio module instances a, b and c in the first audio path and the 2 audio module instances x, y in the second one:

    [[inst_module_a, inst_module_b],[inst_module_b, inst_module_c],
 [inst_module_x, inst_module_y]]

The following shows a JSON audio configuration file that contains the three audio scenario definitions promptingScenario, recognitionScenario and combinedScenario. The promptingScenario audio scenario contains 1 audio path connecting a Prompter audio module instance named prompter with an Audio Output audio module instance named loudspeaker. The recognitionScenario audio scenario contains 1 audio path connecting an Audio Input audio module instance named microphone with a Speech Signal Enhancement audio module instance named sse which in turn is connected with with an Asr audio module instance named recognizer. The combinedScenario audio scenario contains 2 audio paths illustrating a combination of the audio path definitions of the promptingScenario and recognitionScenario audio scenario in one single scenario definition.

Audio Modules

Description

The delivery contains a set of so called audio modules. Each audio module provides a specific audio processing capability (e.g. speech recognition). To realize complex audio use cases these audio modules can be chained together in so called audio scenarios that can be activated and deactivated. For more details see Connecting Audio Modules.

Furthermore there exist an implicit and an explicit method to create and destroy audio modules. For more details see Audio Module Creation and Destruction.

Subcomponents for Audio Modules

Audio Adapters

Description

The audio framework uses an audio adapter approach to realize the access to external audio sinks and sources of any kind whatsoever (e.g. audio devices, pipes, ..) as explained in the basic concepts chapter External Audio Integration. This is realized with the Audio Input and Audio Output audio modules of the SDK which use audio adapters to perform external audio I/O. Thereby each instance of an Audio Input and Audio Output module can be configured to use a specific audio adapter and thus to communicate with a specific external audio data sink respectively source. This approach offers the possibility to define complex audio module scenarios with external audio I/O in a flexible manner. To keep the Audio Input and Audio Output modules independent of any specific adapter implementation, each audio adapter needs to implement a generic adapter interface.

The audio framework of the SDK is capable to handle any number of audio adapters. Therefore each adapter is registered with a unique name (e.g. asio_audio_adapter), the so called adapter type, to the internal adapter registry of the Audio Manager. This adapter type name can be chosen individually by the developer of an adapter implementation, but has to be unique within its adapter category (input resp. output adapter). The JSON configuration of an Audio Input and Audio Output module instance contains an adapter_type parameter to have the possibility to bind a specific adapter implementation to it. With this information the Audio Input and Audio Output module instances are capable to request an instance of the corresponding adapter at the internal adapter registry of the Audio Manager and to use it then via the generic adapter interface.

In fact each audio adapter implementation consists of an audio adapter factory and the actual audio adapter implementation. The audio adapter factory implements the generic nuance_audio_IAudioInputAdapterFactory or nuance_audio_IAudioOutputAdapterFactory interface and with that provides the possibility to create instances of the specific audio adapter implementation on request. This API is used by the Audio Manager when an Audio Input or Audio Output module request an instance of a specific adapter_type. In contrast the audio adapter implements the generic adapter interface nuance_audio_IAudioInputAdapter or nuance_audio_IAudioOutputAdapter which is used by the Audio Input or Audio Output module to perform audio I/O with the adapter and so with the external audio sink or source.

In order that it can be used each audio adapter factory needs to be registered at the Audio Manager with the procedure described in chapter Registering External Audio Adapters.

Sample Implementations

Sample implementations of an input and an output audio adapter are contained in the SDK delivery in the SDK sample programs.

API Reference

The API reference documentation of the audio adapters are part of the audio core component and are included in the Cerence SDK package at doc/api_reference/audio/audio_core/. There is a separate folder for each supported programming language in which the reference documentation can be accessed via the main page index.html.

The C API documentation is available at doc/api_reference/audio/audio_core/c/index.html.

Configuration API Reference

In general for each audio adapter implementation a specific JSON configuration can be defined. This configuration can be set at the corresponding Audio Input or Audio Output module instance configuration via the adapter_params whose JSON string is given one-to-one from the Audio Input or Audio Output module instance to the audio adapter implementation. Needless to say that the interpretation and JSON parsing needs to be done by the adapter itself.

See also the JSON configuration API of the Audio Input or Audio Output described in chapter Audio Input Configuration API Reference and Audio Output Configuration API Reference.

Note

As audio adapter implementations are highly platform and project specific the SDK delivery does contain only sample audio adapter implementations for each standard platform SDK package. These sample adapters are provided as source code and may serve as a starting point for Cerence PS or customer engineers to develop project specific adapter(s). Note that the sample adapters are not guaranteed to work on each specific platform. Finally it is the responsibility of Cerence PS or the customer to implement and test a properly working external audio integration in the form of an audio adapter.

Description

Thereby each instance of an Audio Encoder or Audio Decoder audio module can be configured to use in addition to a codec a specific audio container. To keep the Audio Encoder and Audio Decoder modules independent of any specific container implementation, each audio container has to implement a generic container interface.

With this information the Audio Encoder or Audio Decoder module instance is capable to request an instance of the corresponding container at the internal container registry of the Audio Manager and to use it then via the generic container interface.

Glossary

Audio Path
An audio path is a concatenation of audio modules to realize an audio processing chain for a specific use case. Thereby the very first audio module must have an audio output (audio source) only, the very last an audio input (audio sink) only and all in-between at least one audio input and one output.
Audio Scenario
An audio scenario is a definition of one or more audio paths. Audio scenario can be defined with a unique name in the JSON audio configuration file. Each audio scenario can be activated and deactivate by usage of its unique name via the Audio Manager API.
