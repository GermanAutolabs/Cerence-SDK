ISpeechSignalEnhancementConfig - The configuration of the speech signal enhancement module.

Super type: IAudioModuleConfig

The configuration of the speech signal enhancement module.

The ISpeechSignalEnhancementConfig interface is used to represent the configuration of the speech signal enhancement module.

The speech signal enhancement module integrates the sse core component into the SDK’s audio backbone, if configured. Depending on its configuration it will do beam forming, echo cancellation, etc.

Details:

"name":

The name of the audio module.

This name has to be unique. For instance it is used to reference this module from audio scenarios (audio paths).

"sse_lib":

The name of the sse shared library file.

With the parameter "sse_lib", the name of the shared object containing the sse core component used in the speech signal enhancement module is being configured. The name must be in the format "sse_major_minor", where "major_minor" are optional. e.g. "sse_4_1" or "sse_4_2" or "sse" and must not include platform-specific prefixes (like “lib”) or suffixes (like “dll”, “so”) or file path information. Any name value that does not match that format will lead to an error.

By default the speech signal enhancement module uses the latest sse library which is part of a standard delivery package ("sse").

SSE versions that are older than the latest offered version are supported under following circumstances:

  • The library file name must be in the above described naming format.
  • The library file must be available at the expected location in the file system.
  • The library file contains an SSE 4.1.4 or more recent.

If it is required that a speech signal enhancement instance uses an older sse library version than the default one, the relevant sse library name must be configured in the above described format. E.g. if the version of the older library is 4.1, the library must be named <prefix>sse_4_1.<suffix> and the value of the "sse_lib" property must be defined identically to the library part name "sse_4_1". Note, that the speech signal enhancement module checks during loading of the sse library if the passed version information matches. In case of a mismatch the creation of the speech signal enhancement instance fails.

Examples showing how to use an sse version 4.1 on different platforms:

  • On Windows: sse library is named sse_4_1.dll and the property defined to "sse_lib": "sse_4_1"
  • On Linux: sse library is named sse_4_1.so and the property defined to "sse_lib": "sse_4_1"
  • On Android: sse library is named libsse_4_1.so and the property defined to "sse_lib": "sse_4_1"

Note: the parameter "sse_config_file" needs to be set accordingly. That means for an "sse_lib" referencing an sse 4 a “*.scd” file needs to be configured.

"sse_config_file":

The name of the sse configuration file.

With the parameter "sse_config_file", the main configuration file for the sse core component inside of the speech signal enhancement module is being configured.

If the parameter "sse_config_file" is not defined, a build in single channel configuration for cloud recognition is activated.

Note: The audio format of the modules around the speech signal enhancement module must match the configured values in the sse configuration file in terms of channel_count and sample_rate. The speech signal enhancement module will use the same amount of samples_per_channel at the output as received via the input, independently of the configuration within the sse configuration file.

The property supports the vh_callback mechanism, meaning that the sse_config_file can be changed during runtime.

Example: “sse_config_file”: { “vh_callback”: { “name”: “(instance_name)_sse_config_file”, “type”: “string”, “default”: “” } }

it is proposed to expand/replace (instance_name) with the name property of the module, defined in the IAudioModuleConfig, to ease log analysis and ensure unique dynamic names.

The SpeechSignalEnhancement module evaluates the sse_config_file each time the scenario it is used in is prepared (in more detail when the module is instantiated).

The default parameter in the vh_callback is optional and can be set if the user sees a need for it. Setting the vh_callback default value makes the vh_callback to return always a valid value.

If this vh_callback default value is not set and nothing is set from outside the vh_callback will return nothing. In that case the property sse_config_file is handled like not defined at all and the sse_config_file default value “” is used.

"adapt_on_connect":

The flag to activate immediate data capturing after being connected.

This parameter should be set only if the effects described below are understood and wanted. If unsure do not set this value.

When this parameter is set to true, the instance of the speech signal enhancement module requests immediately audio data from its connected input (source_pad) after the connection to the audio backbone is being established.

This is the case once an audio scenario which is using this speech signal enhancement module is activated. The captured audio data is only forwarded if requested by a following module (e.g. recognizer), otherwise it will be used for internal adaption only.

If the following module in the backbone indicates that no more data is needed, the speech signal enhancement module stops delivering, but does not forward that request to the modules connected to the speech signal enhancement’s input. This way the speech signal enhancement module will continue capturing audio data silently to keep an uninterrupted audio stream. The default value is false. The parameter should be set only if there is a well-founded knowledge about the SDK’s audio framework. If unsure, do not change the parameter.

Note: If this parameter is activated, the prompt synced input module has to be placed behind the speech signal enhancement module within the audio backbone. The file list feature of audio from file will not work, and the audio from file module may not be usable at all, since data is consumed instantly when activating the audio scenario.

"signal_statistics_interval":

The interval (in milliseconds) how often onSseSignalStatistic() is called.

This value needs to be configured if the desired signal statistic data should be emitted. If this value is not defined no signal statistic data are send. For SDK users the emission is done via the audio managers “onSseSignalStatistic” listener method. For CDFW users a corresponding GCF event is generated.

The value 0 has a special meaning. It will activate the sending of the event at the end of each incoming stream only. Please be aware that with adapt_on_connect activated there will be only one incoming stream.

We recommend values above 1000ms to get meaningful arithmetic mean values and have a low system load.

The property supports the vh_callback mechanism, meaning that the signal_statistics_interval can be changed during runtime.

Example: “signal_statistics_interval”: { “vh_callback”: { “name”: “(instance_name)_signal_statistics_interval”, “type”: “integer”, “default”: -1 } }

it is proposed to expand/replace (instance_name) with the name property of the module, defined in the IAudioModuleConfig, to ease log analysis and ensure unique dynamic names.

The SpeechSignalEnhancement module evaluates the signal_statistics_interval each time the scenario it is used in is activated (in more detail when its input is connected).

The default parameter in the vh_callback is optional and can be set if the user sees a need for it. Setting the vh_callback default value makes the vh_callback to return always a valid number.

If this vh_callback default value is not set and nothing is set from outside the vh_callback will return nothing. In that case the property signal_statistics_interval is handled like not defined at all and the signal_statistics_interval default value “-1” is used.

"active_acoustic_zones":

The acoustic zones the sse uses to build its mixed output signal.

This configuration parameter is required for the zone suppression feature of the sequential MultiZone UseCase.

For this feature it is necessary to define which zones go to the mixed data stream for the recognizer doing the wake up word recognition (e.g. asr_0).

This configuration parameter influences the first sse output channel only.

The property supports the vh_callback mechanism, meaning that the active acoustic zones can be changed during runtime.

Example: “active_acoustic_zones”: { “vh_callback”: { “name”: “(instance_name)_active_acoustic_zones”, “type”: “array”, “default”: [] } }

It is proposed to expand/replace (instance_name) with the name property of the module, defined in the IAudioModuleConfig, to ease log analysis and ensure unique dynamic names.

Note: It is recommended, that no recognition is running when this configuration parameter is changed.

The SpeechSignalEnhancement module evaluates the active_acoustic_zones as follows:

  • when the module is created
  • when the subsequent module requests data (in more detail when its output starts a session, like done when starting the recognizer with VoConHigh 5)
  • when data is processed every 250ms

The default parameter in the vh_callback is optional and can be set if the user sees a need for it. Setting the vh_callback default value makes the vh_callback to return always a valid value.

If this vh_callback default value is not set and nothing is set from outside the vh_callback will return nothing. In that case the property active_acoustic_zones is handled like not defined at all.

"reset_timing":

Define what type of reset has to be done between two successive audio streams.

This value defines a list with two elements (A,B). Each defines a timespan in ms. The value A has to be less or equal to the value B otherwise the SSE module will refuse to load that configuration value and abort creation. Additionally to that the values 0 and -1 are permitted. With 0 the mentioned reset type is performed always without evaluating any timestamps. With -1 the mentioned reset type is disabled and not performed.

Inside of the speech signal enhancement module the time passed between two successive incoming audio data streams is measured, if needed to determine which reset type has to be performed.

Two values are defined with that configuration parameter A (the first value) and B (the second value).

  • If the measured timespan is greater than B a full reset is done
  • If the measured timespan is greater than A but smaller than or equal to B a partial reset (type 2) is done
  • If the measured timespan is smaller than or equal to A nothing is done

The internal default values are 0 for A and 5000 for B. That means the following: If the measured timespan is greater than the B’s 5000ms a full reset is performed otherwise a partial reset (type 2) is performed always at the beginning of each incoming data stream.

Those default values were agreed with the acoustic team building the Speech Signal Enhancement (SSE) core library. There is no general need to change those configuration values. This documentation does show only that the possibility exists. If unsure please let the defaults in place.

Note: Setting those values to 0,0 or -1,0 triggers a full reset at the beginning of each data stream. Please check very carefully if that is really intended.

Note: Setting those values to 0,-1 triggers a partial reset (type 2) at the beginning of each data stream. Please check very carefully if that is really intended.

Note: Setting those values to -1,-1 prevents full or partial type 2 resets at the beginning of each data stream. That means that there are no resets at all. Please check very carefully if that is really intended.

Note: If adaptOnConnect is activated, the speech signal enhancement module will have one audio data stream at the input only as long as the audio scenario is active.

"activate_tuning_port":

parameter used to activate the tuning port

The speech signal enhancement module provides an interface to access and tune the sse core component instance during runtime. The high level interface opens a TCP server on a configurable port to connect with the SSE/ICC Tuning Tool.

Since the tune commands are sent to the sse core component instance together with incoming audio data, it is vital for the tuning interface that audio data is flowing.

Therefore see also parameter "adapt_on_connect".

The tuning interface can simply be activated by setting this parameter to a TCP port number (integer) greater than zero. The standard TCP port for the SSE/ICC Tuning Tool is 2012.

"prompt_start_info_trigger":

parameter used to define when to notify the sse core component instance that a prompt is going to be played

This value defines when to notify the sse core component instance that a prompt is going to be played.

This value shall only be changed following direct recommendations of the acoustic expert team.

If unsure do not change.

" promptStartInfoType ": define the prompter information type. This enumeration defines the different prompter information types which can be used to notify the sse core component instance that a prompt is going to be played

  • "PROMPT_FIRST_WRITE": the 1rst chunk of a prompt left the audio output
  • "PROMPT_FIRST_CONFIRMATION": the confirmation that the 1rst chunk of a prompt was processed, reached the audio output

IAudioScenarioConfig - The audio scenario configuration.

Super type: ITopLevelConfig

The audio scenario configuration.

The IAudioScenarioConfig interface is used to represent the configuration of an audio scenario, i.e. the definition of a set of audio paths (source-sink connections) that can be activated together using the audio manager.

An audio scenario configuration needs to define two items.

  • First, a unique name. This name needs to be used to activate the audio scenario during runtime via IAudioManager.activateScenario(name).
  • Second, an audio path definition, which is a list of source sink connections defined in an array. For such a connection a list with two elements is used. The first element is the name of an audio source, the second element the name of an audio sink.

The array of source-sink lists is building the “audiopaths” configuration parameter. Imagine there is a producer named “mod1”, a processor named “mod2” and a consumer named “mod3”. The definition of the audio path connecting those three modules would look as follows:

"audiopaths": [ ["mod1", "mod2"], ["mod2", "mod3"] ]

The names in that list must match the ones used in the application program when creating a modules instance or refer to a built-in module implementation. Let us assume, for example, that an application creates a prompter instance with the name myprompter:

nuance.prompter.IPrompter.create("myprompter", config, errorListener)

Then this same name myprompter needs to be used in the audio JSON configuration file to define the audio path, for example:

"audiopaths": [ ["myprompter", "default_speaker"] ]

This example defines an audio path where the myprompter source module produces audio data that shall be passed to the consumer module default_speaker which is here supposed to refer to the built-in AudioOutput implementation. This built-in module will be started automatically by the audio manager with the name "default_speaker" if configured as shown and not already present.

Each audio scenario must define at least one source sink connection, using valid module names. Given these audio path definitions the audio manager will set up the required connections for the audio data flow between the modules. Thus, the possibility to define different audio scenarios allows a very flexible configuration of audio processing chains within the audio manager.

Only configured audio scenarios can be activated at the audio manager’s interface. Handing an unknown scenario name will lead to an error.

Key Name Value Type Required? Documentation
"name": String yes The name of the audio scenario.
"audiopaths": [ [ String ] ] yes The audio paths of this audio scenario.

Details:

"name":

The name of the audio scenario.

This name is used for referencing this audio scenario when activating it through the audio manager API.

"audiopaths":

The audio paths of this audio scenario.

An audio path is defined as a list of 2-element lists.

The following naming scheme is used to define a connection in that 2-element lists.

Each sink or source pad is referenced by the name of its module, then followed by a colon (“:”) and the name of the sink or source pad itself. The syntax is module_name:source_pad_name respectively module_name:sink_pad_name.

Note: The colon delimiter therefore cannot be used as part of any module name, sink pad name or source pad name.

For modules having just one sink and/or source pad the name of the sink or source pad does not have to be defined, since the modules name is already a unique key then.

For more complex modules with multiple sink and source pads the audio path configuration needs to use the colon delimiter syntax module_name:source_pad_name respectively module_name:sink_source_name.

IAudioModuleConfig - The audio module configuration.

Super type: ITopLevelConfig

The audio module configuration.

The IAudioModuleConfig interface is used to represent the generic configuration of audio modules. The derived audio module (configurations) inherit the name property defined here.

It is possible to add dedicated module configurations to the audio manager’s configuration file.

This is important if the built-in “AudioInput”/”AudioOutput” should be used, for instance. Since a module is only ready for automated internal startup, if it is one of the “known modules” listed below and configured in the modules section.

Adding configurations for other module types than the “known modules” listed below will cause an error while loading this configuration.

  • AudioBuffer
  • AudioDecoder
  • AudioEncoder
  • AudioFromFile
  • AudioInput
  • AudioOutput
  • PromptSyncedInput (FW only)
  • AudioSegmentation
  • AudioSignalAnalysis (FW only)
  • SampleRateConverter
  • SpeechSignalEnhancement
  • AudioStreamDistributor
  • AudioToFile

Currently known module types are listed as “sub types”, see the documentation of the sub-types for the list of type-specific properties.

Note: Configuring modules of those types and referencing their names in the audio path of a scenario will allow the implicit creation of a module instance if no module with that name is present so far. The implicit creation is done during the preparation step of an audio scenario activation. Please choose the names for your modules carefully to avoid name conflicts.

IChunkPoolConfig - ChunkPool configuration.

ChunkPool configuration.

Key Name Value Type Required? Documentation
"chunk_count": int no, default value is 100 The initial number of chunks per audio format that are allocated.
"chunk_realloc_count": int no, default value is 50 The number of audio chunks used for reallocations.
"chunk_threshold": int no, default value is 2024 Threshold for dynamic versus static allocation within an audio chunk.
"marker_count": int no, default value is 100 The initial number of audio markers that gets allocated.
"marker_realloc_count": int no, default value is 100 The the number of audio markers used for reallocations.

Details:

"chunk_count":

The initial number of chunks per audio format that are allocated.

This number of audio chunks of a certain audio format gets allocated the first time a reserve is made for this audio format. The number of allocated chunks will never fall below this value, even if all reservations are cleared.

"chunk_realloc_count":

The number of audio chunks used for reallocations.

This number of audio chunks of a certain audio format gets allocated when the number of reservations exceeds the current number of allocations. The same number is also used for decreasing the number of allocated chunks when reservations are cleared.

"chunk_threshold":

Threshold for dynamic versus static allocation within an audio chunk.

This value only takes effect for chunks carrying encoded audio data. If the chunk size is below the threshold it is tried to re-use already allocated memory while bigger audio chunks are allocated dynamically. For PCM audio data a complete static approach is used.

"marker_count":

The initial number of audio markers that gets allocated.

This number of audio markers gets allocated the first time a marker is requested.

"marker_realloc_count":

The the number of audio markers used for reallocations.

This number of audio markers gets allocated when an audio marker is requested and no more unused marker is available.
