Like any other component, the common component can be configured through JSON configuration files. The following section lists all available configuration keys.

Configuration Parameters

ILoggerConfig - The logger configuration.

The logger configuration.

Key Name Value Type Required? Documentation
"text_buffer_entries": int no, default value is 1024 The number of text messages to buffer
"data_buffer_size_per_channel": int no, default value is 16384 The size of the buffer for binary log data
"log_WARNING_INFORMATION_on_ERROR": String, one of "true" and "false" no, default value is “false” Flag for including additional recent messages
"text_filters": [ { ILoggerTextFilterConfig } ] no, default value is <empty> The filters for excluding certain log messages
"dlt": { IDltConfig } no The DLT (Diagnostic Log and Trace) definitions
"consumers": [ { ILoggingConsumerConfig } ] yes The logging consumer configurations (at least one consumer must be defined).

Details:

"text_buffer_entries":

The number of text messages to buffer

To decouple logging consumers from logging producers the size of the used buffers can be influenced by the parameter “text_buffer_entries” and “data_buffer_size_per_channel”. “text_buffer_entries” is not a size in bytes. It just defines how many text messages are kept.

"data_buffer_size_per_channel":

The size of the buffer for binary log data

“data_buffer_size_per_channel” specifies the size of the buffer for binary log data in bytes. Each consumer has its own buffer, i.e. if one slow consumer is causing a buffer overrun, this has no effect on other consumers.

"log_WARNING_INFORMATION_on_ERROR":

Flag for including additional recent messages

If we have logging consumers for the zones LOG_FATAL and LOG_ERROR only, it’s sometimes hard to find the root cause of a bug, if the history of the logging for the zones LOG_WARNING and LOG_INFO is missing. To avoid this the parameter “log_WARNING_INFORMATION_on_ERROR” can be set, by this, recent messages got before and having the zone LOG_WARNING or INFO are logged additionally too after there was an LOG_ERROR or LOG_FATAL. Since logged lines do have a time stamp, sorting by time can be done by a post processor. To be able to-do so, the buffer used for text logging is kept always 2/3 filled. Then by getting an LOG_FATAL or LOG_ERROR, all text LOG_WARNING and LOG_INFO messages which happened before and which are still on the buffer are directly logged. Having the buffer 2/3 filled also if they have been processed by the thread who serves the consumers does not have an impact regarding a log buffer overrun. That’s because when there are new messages, at this time the are removed to keep the buffer 2/3 filled.

"text_filters":

The filters for excluding certain log messages

The parameter “text_filter” can be used to filter out messages starting with a special text. A filter can be set to hide known issues for customers.

"dlt":

The DLT (Diagnostic Log and Trace) definitions

This key contains DLT-specific definitions which all DLT consumers use commonly.

"consumers":
The logging consumer configurations (at least one consumer must be defined).

ILoggingConsumerConfig - The logging consumer configuration.

The logging consumer configuration.

The ILoggingConsumerConfig interface is used to represent the configuration of log consumers, i.e. objects that receive log messages from the central logger singleton and store it in files etc.

Key Name Value Type Required? Documentation
"name": String yes The name of the consumer.
"output": String no The place to store the log output.
"max_file_size": int no, default value is -1 Optional parameter given in bytes to limit the file system space used by the log files.
"max_files_number": int no, default value is -1 Optional parameter to limit the number of log files which contain the placeholder.
"enable_backup_log": String, one of "true" and "false" no, default value is “false” Optional parameter to keep log files from previous runs.
"modules": String no, default value is <empty> The modules to log.
"not_modules": String no The modules NOT to log (optional, default: “”).
"zones": String yes The logging zones for which log output should be collected.
"params": { IUserJsonConfig } no Contains consumer params which are specific to the consumer type.

Details:

"name":

The name of the consumer.

If user-defined log consumers are registered at the runtime API (not supported in CDFW), the name is used to identify the corresponding configuration entry for a registered consumer. Obviously this name must be unique.

"output":

The place to store the log output.

If this attribute contains stdout or stderr the log messages are written to standard output / error.

If it contains TRACECLIENT: followed by a file path, the log messages are written to an external Trace Client.

If it contains DLT the log messages are written to DLT (Diagnostic Log and Trace).

If it contains SOCKET the log messages are written to the connected client over the TCP/IP socket. This type of the output configuration has to be defined in the following format:

SOCKET:port='server port';buffer='size in bytes'.

Parameter port is required and it contains the port number that the socket log consumer will use to create server socket and wait for the client connection. Only one client connection at a time is supported. Parameter buffer is optional and it specifies the size in bytes the socket log consumer will use to buffer the messages (or data). If buffer of size 0 is configured then no buffering will take place; instead logged content is sent directly to the socket if a connection is present. The default value for the buffer size is 16384 bytes.

If it contains PLUGIN: followed by a plugin’s shared library file name, the log messages are written to an external plugin logging consumer implementation. The file name may contain an absolute (i.e. starting with / or [A-Za-z]:[/]), a relative or none path information. In the latter two cases the platform specific search strategy (e.g. LD_LIBRARY_PATH on Linux) is used. In case a file name does not contain a platform specific shared object prefix (e.g. lib) or extension (e.g. .so), it is automatically added. Examples:

  • full qualified plugin file name: "output": "PLUGIN:/usr/lib/lib_plugin_log_consumer.so"
  • full qualified plugin file name without shared library prefix and extension: "output": "PLUGIN:/usr/lib/plugin_log_consumer"
  • without path information, using search strategy of platform, e.g. LD_LIBRARY_PATH on Linux: "output": "PLUGIN:pluggable_file_consumer"

In case more than one instance of the same plugin library is configured the name configuration parameter must be each unique.

If none of these patterns match, output is interpreted as a file path. The file path can contain an absolute path to the file, or a path relative to the JSON configuration directory. To be able to avoid that files are overwritten, the file name can be extended with the following placeholders:

  • %DATE_TIME% is replaced by a time stamp that contains the current date and time (format: YYYY-MM-DD_HH-MM-SS). This is the file creation time.
  • %TIME_MS% is replaced by a time stamp that contains the current time since the process started (format: HH-MM-SS-MS). This is the time stamp of the 1rst entry written into the file.

The output parameter is optional. If it is omitted, the framework does not create a log consumer automatically, but the user can add a consumer implementation at runtime, using ILogging.addConsumer().

"max_file_size":

Optional parameter given in bytes to limit the file system space used by the log files.

Default value for this parameter is -1 meaning no size limitations. When the parameter “max_file_size” is set to the value greater then zero SDK (Logger module) will first verify the logging zone types that are configured for this file. If log zone is configured to “LOG_DATA” then the file is handled like a ring buffer where the logging module seeks to the beginning of the file when the size limit is reached and starts logging a new content from there.

If textual zone(s) are configured and the parameter “max_file_size” is set then SDK will use rotating log files mechanism to ensure that not more then defined maximal size is used on the file system for the logging. SDK currently uses up to 4 log files for the rotation and the maximal size per one file is computed as “max_file_size” value divided by 4. The reason for splitting up into several files is to avoid that large portions of the log history are lost when maximum file size is exceeded and the new log file has to be opened. Logger will always log current (most recent) content to the log file with the name defined with the “output” parameter. For example, let’s assume that the file name is “voconhigh.log”. When the computed size limit per one file is reached then the file is closed and renamed to have the extension “.1” after the original name, for example “voconhigh.log.1”. New logging continues to the newly opened file with the original name, f.i. “voconhigh.log”. When the next limit is reached, again new “voconhigh.log” is opened for the new logging and the previous log content is shifted like presented in the following scheme: “voconhigh.log” -> “voconhigh.log.1” -> “voconhigh.log.2” -> “voconhigh.log.3”. As a result we always have at most 4 files that in-total occupy at most “max_file_size” of the file system space. For an easier analysis later one can concatenate these files manually or with some tool in a reverse order (.3->.2->.1->original file) and restore one log file that contains most recent “max_file_size” bytes of the logged content. If “max_file_size” value is configured to be below 64K bytes (65536) then SDK will set “max_file_size” parameter automatically to 65536 bytes to prevent that rotation files contain too small fragments of the logged content. The value of “max_file_size” must not be set to greater than 2147483647.

"max_files_number":

Optional parameter to limit the number of log files which contain the placeholder.

Values in the range [-1,100] are possible. The default value for this parameter is -1 meaning no limitation of the number of log files. The value specifies the maximum number of log files which are kept on disk and which match the file name pattern defined with the “output” key. If the maximum number of files on disk is reached and a new log file is opened the oldest will be removed. Log files from previous startups are taken into account, i.e. they are kept on disk as long as the maximum is not exceeded. In combination with max_file_size the occupied disk space for this consumer will be limited to max_file_size * max_files_number.

"enable_backup_log":

Optional parameter to keep log files from previous runs.

Default value for this parameter is “false” meaning an existing log from the former run will be deleted.

If the value for this parameter is set to “true” the existing log on the file system will be continued on the last position. With that an overall log will be created for all former startups / sessions.

This configuration parameter can be used in combination with max_file_size, meaning that the overall log will be split into 4 log files of limited size.

"modules":

The modules to log.

Logging information is collected only for the modules specified here. Use “|” as delimiter when listing multiple modules.

"not_modules":

The modules NOT to log (optional, default: “”).

This value can be used to suppress the logging output of certain modules. Use “|” as delimiter when listing multiple modules.

"zones":
The logging zones for which log output should be collected.
"params":

Contains consumer params which are specific to the consumer type.

This JSON string key can be used to pass specific configuration parameters to a logging consumer. The string has to be in JSON format. The JSON format must follow a defined json structure which depends on the type of the logging consumer (e.g. DLT, PLUGIN). See the below list showing which logging consumers support the params key and where to find the corresponding json structure:

  • DLT: Look at IDltConsumerParamsConfig
  • PLUGIN: Depends on the external plugin logging consumer implementation.

IThreadParametersConfig - The configuration of thread parameters.

The configuration of thread parameters.

This configuration class defines the parameters that can be used to configure a specific thread (referenced by name).

Key Name Value Type Required? Documentation
"name": String yes The thread name.
"priority": int yes The thread priority.
"policy": " THREAD_SCHED " yes The scheduling policy.
"sporadic_scheduling_params": { IThreadParametersSporadicConfig } no The parameters to configure Sporadic Scheduling
"adaptive_partitioning_params": { IThreadParametersAdaptiveConfig } no The parameters to configure Adaptive partitioning.

Details:

"name":

The thread name.

The name "default" has a special meaning, it is used to set default values for “priority” and “policy”. At the moment it is not possible to set default values for Sporadic Scheduling and adaptive partitioning parameters. Note: If the default for scheduling policy is set to "other" then the default for "priority" has to be set to 0 since this is currently the only supported value.

"priority":

The thread priority.

0 is the “normal” priority. Use positive numbers for high priority threads, negative numbers for less critical threads. The minimum/maximum priority is platform-dependent. Note: If scheduling policy is set to "other" then the priorities are implemented by setting a nice level. This means that negative values mean high priorities in this case.

"policy":

The scheduling policy.

" THREAD_SCHED ": The scheduling policy. Not all policies are available for all platforms.

  • "unknown": used on platforms unaware of scheduling policies
  • "fifo": First-in, first-out: threads are queued in the order they arrive in the ready queue
  • "rr": fixed time per thread, scheduler cycles through them
  • "sporadic": Sporadic Scheduling, only available for QNX
  • "batch": batch scheduler, available for Linux and Android
  • "other": platform-specific scheduler
"sporadic_scheduling_params":
The parameters to configure Sporadic Scheduling
"adaptive_partitioning_params":
The parameters to configure Adaptive partitioning.

IThreadParametersSporadicConfig - The configuration of the Sporadic Scheduling policy.

The configuration of the Sporadic Scheduling policy.

Parameters to configure Sporadic scheduling. For details see QNX documentation.

Key Name Value Type Required? Documentation
"low_priority": int yes The low priority.
"max_repl": int yes The maximum number of pending replenishments.
"init_budget": int yes The initial budget.
"repl_period": int yes The replenishment period.

Details:

"low_priority":

The low priority.

The priority level to which the thread will drop. The thread executes at this lower priority while in the background, and runs at normal priority while in the foreground.

For details see QNX documentation.

"max_repl":

The maximum number of pending replenishments.

This value limits the number of replenishment operations that can take place, thereby bounding the amount of system overhead consumed by the Sporadic Scheduling policy.

For details see QNX documentation.

"init_budget":

The initial budget.

The amount of time a thread is allowed to execute at its normal priority before being dropped to its low priority.

For details see QNX documentation.

"repl_period":

The replenishment period.

The period of time during which a thread is allowed to consume its execution budget. To schedule replenishment operations, the POSIX implementation also uses this value as the offset from the time the thread becomes READY.

For details see QNX documentation.

Defining Variables in JSON Configurations

The common JSON configuration offers the possibility to define variables. This can be used, for example, to accumulate all platform specific configuration parts in a single JSON file and keep all other configuration files generic. This can be achieved using the “variable” mechanism. The platform specific file needs to define an array named "json_variables", that defines variables in a “name / value” scheme, where the "value" field can hold any valid JSON element. These variables can then be used in any other JSON file by referencing that variable: {“variable” : “var1”}. The whole object including the brackets will be replaced by the JSON element defined in the "value" field. The variable name needs to be a unique string.

Example:

{
  "version" : "Common 1.0",
  "json_variables": [
    {
      "name"  : "v_path",
      "value" : "./path/to/vocalizer"
    }
  ]
}

{
  "version" : "Prompter 1.0",
  "prompter": [
    {
      name": "LocalPrompter",
      "type": "local",
      "local": {"vocalizer_path": {"variable" : "v_path"}}
    }
  ]
}
