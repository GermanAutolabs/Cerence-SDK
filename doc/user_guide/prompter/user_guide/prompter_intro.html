Types of the Prompter

The Prompter supports different IPrompter instance types. There is the local type that is based on the Cerence TTS Embedded engine. Additionally there is the cloud type providing TTS via the Cerence Cloud. Both types have the same API. Important to mention is that the local type supports the full set of the provided API methods. The cloud type currently supports a reduced set of the methods. If a method is not supported it will return an error result.

Using the Prompter

The main Prompter interface is the IPrompter class. Each IPrompter instance has an own internal thread which is used to execute operations asynchronously in the background. This thread has a queue, from which it reads and executes background operations sequentially. The IPrompter class provides a couple of asynchronous methods like nuance_prompter_IPrompter_activateUseCase, nuance_prompter_IPrompter_setParameter, nuance_prompter_IPrompter_setStringParameter or nuance_prompter_IPrompter_analyzeText to start operations on an IPrompter instance (see more details regarding the methods under Runtime Activation of a Usecase and Advanced Prompter API Functions). IPrompter provides the nuance_common_IBackgroundOperation interface which allows to control those Prompter-related operations, e.g. to wait for completion or to abort them. To wait for the IPrompter instance to complete its queued background operations the following synchronous method is used:

nuance_prompter_IPrompter_waitForCompletion

This call also provides an output parameter status to test the success of the operation.

To abort all queued/executing Prompter-related background operations abortOperation has to be called:

nuance_prompter_IPrompter_abortOperation

If an instance of an nuance_prompter_IPrompterListener implementation is registered at the used IPrompter instance (either at IPrompter creation time or with nuance_prompter_IPrompter_setListener), the application can listen to warnings, errors, or various results or states reported during Prompter-related operations.

Runtime Activation of a Usecase

In order to be able to read out text with an IPrompter instance, a usecase configuration must be activated on the instance. A usecase defines, among other things, the voice used for text synthesis. Usecases are defined in form of JSON configurations and must be provided to the IPrompter instance already at creation time. The activation of a certain usecase is accomplished with the API call:

nuance_prompter_IPrompter_activateUseCase

It takes as input parameter the name of the usecase to be activated which must match a usecase defined in the Configuration.

After calling nuance_prompter_IPrompter_activateUseCase the activation happens asynchronously in the background. Returning of the method does not mean that the actual usecase activation has finished. To wait for the completion of the usecase activation the call nuance_prompter_IPrompter_waitForCompletion has to be used. To check the success, the output parameter status of nuance_prompter_IPrompter_waitForCompletion can be used.

Additionally the application can listen for warnings and errors with a registered nuance_prompter_IPrompterListener implementation.

During activation of a usecase it is ensured that all optional properties that are not configured in the JSON files associated to this usecase are set to default values. For example if the current activated usecase has loaded specific tuning_files, an active_prompt_domain, or a text_normalization_type but the new usecase does not contain any of those properties the system will unload and deactivate them. If the new usecase has different tuning_files defined the old ones are unloaded and the new ones are loaded. The same rules apply if the associated voice_config of the current activated usecase has defined several optional properties like e.g. volume, readmode and lidscope but the voice_config of the new usecase does not define them, then the system will restore them to the default values of Cerence TTS Embedded. Note this does not apply for properties name, voice, operating_point and language as they are not optional and must be anyway defined in each voice_config.

Errors occurring during an activation of a usecase will cause the background operation to stop and to return an error status.

The nuance_prompter_IPrompter_waitForCompletion returns status = NUANCE_COMMON_IBACKGROUNDOPERATION_FAILED. An error is sent to nuance_prompter_IPrompterListener_onError.

In the error-case the IPrompter re-activates the usecase that was set before. The instance remains “unconfigured” if no usecase was active before. No prompting is possible in this state.

Warnings happening during activation of a usecase do not cause the background operation to stop.

Instead warnings are sent to nuance_prompter_IPrompterListener_onWarning and the operation continues.

A warning indicates that optional parts of the given usecase failed to be activated. They may be caused if, for instance, the tuning files or one or more optional voice_config parameter(s) could not be set or configured.

Playing Prompts

The main API methods of IPrompter are the methods:

nuance_prompter_IPrompter_playString
nuance_prompter_IPrompter_playFile

which do TTS conversions of a string or a text file, respectively. Both calls return an IPrompt instance which provides an interface to control the running play operation.

For example:

nuance_common_ResultCode rc = NUANCE_COMMON_OK;
const char *inputText = "Hello world!";
nuance_prompter_IPrompter *prompter = 0;
nuance_prompter_IPromptListener *promptListener = 0;
nuance_prompter_IPrompt *prompt = 0;

...

/**
 * At this point, it is assumed that the creations of the IPromptListener and
 * IPrompter instances have already taken place. Further, it is assumed that
 * the IPrompter instance has activated an appropriate usecase.
 */

/* Start the play operation. */
rc = nuance_prompter_IPrompter_playString(prompter, inputText, promptListener, &prompt);
...

The returned output parameter prompt can be accessed with the nuance_prompter_IPrompt interface to control the running operation (e.g. to abort it).

The input text to be read is either in a string or in a UTF8-encoded file. A text string is passed with the text parameter to nuance_prompter_IPrompter_playString. A text file is passed using the filePath parameter to nuance_prompter_IPrompter_playFile. In both cases the input text may contain control sequences to influence the way in which the text is read by the TTS (see Control Sequences). Note that the Prompter passes through the given input text to the underlying TTS engine without any further processing.

It is also possible to pass an instance of a nuance_prompter_IPromptListener implementation to the a play call. The IPrompt instance uses the given listener instance to report errors that may occur during TTS conversion and playback. In addition, the IPrompt instance uses the listener to report its progress and (if enabled in the activated usecase) to inform when certain position markers in the input text are reached.

Both nuance_prompter_IPrompter_playString and nuance_prompter_IPrompter_playFile are asynchronous methods that start playback operations in the background. If a new prompt output is started with one of these play methods, a playback operation is queued for execution by the Prompter’s internal thread and an IPrompt instance is returned. In the same way as the IPrompter class, the IPrompt class provides the nuance_common_IBackgroundOperation interface which allows to control the individual playback operation, e.g. to wait for completion or to abort a running prompt. Usually, an application will wait until a speech output has been completed, before continuing with further processing. This can be achieved by calling the synchronous method nuance_prompter_IPrompt_waitForCompletion after calling one of the play methods. This call has an output parameter status to test the success of the operation.

Destroying IPrompt and IPrompter Instances

After a prompt has finished, it is the application’s responsibility to destroy the corresponding IPrompt instance with the call:

nuance_prompter_IPrompt_destroy

If an IPrompter instance is no longer needed, e.g. at the end of the application, it is the application’s responsibility to destroy all IPrompter instances with the call:

nuance_prompter_IPrompter_destroy
