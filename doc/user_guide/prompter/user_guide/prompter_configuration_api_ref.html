ITopLevelPrompterConfig - The top-level configuration of the prompter component.

The top-level configuration of the prompter component.

Key Name Value Type Required? Documentation
"version": String yes The version key, fixed to “Prompter 1.0”.
"label": String no Label describing the content / version of the JSON file.
"prompter_global": { IPrompterGlobalConfig } no The top-level prompter global configuration.
"voice_data_path": [ { IVoiceDataPath } ] no The configuration of voice data path objects.
"prompter": [ { IPrompterConfig } ] no The prompter configuration.
"usecase": [ { IPrompterUseCaseConfig } ] no The configuration of the prompter use cases.
"voice_config": [ { IVoiceConfig } ] no The configuration of the prompter voices.
"cloud_command": [ { ICloudCommandConfig } ] no The configuration of the prompter cloud command.

Details:

"version":

The version key, fixed to “Prompter 1.0”.

Each JSON configuration file for IPrompter must contain a "version" property. Otherwise the creation of the IPrompter fails and returns an error. The version allows to place different types of JSON configuration files in the same directory. The prompter configuration files will not conflict with configuration files used for a different component than prompter.

"label":
Label describing the content / version of the JSON file.
"prompter_global":

The top-level prompter global configuration.

The JSON configuration may contain a "prompter_global" object, that defines settings relevant to all prompter instances that are defined in the "prompter" list.

"voice_data_path":

The configuration of voice data path objects.

The JSON configuration must consist of one or more "voice_data_path" object(s) if "prompter" objects of type “local” are defined.

For details see section IVoiceDataPath.

"prompter":

The prompter configuration.

The JSON configuration must contain of one or more "prompter" object(s). They can be of different types, like for instance “local” or “cloud”.

For details see section IPrompterConfig.

"usecase":

The configuration of the prompter use cases.

The JSON configuration must contain of one or more "usecase" object(s) They can be of different types, like for instance “local” or “cloud”.

For details see section IPrompterUseCaseConfig.

"voice_config":

The configuration of the prompter voices.

The JSON configuration must contain of one or more "voice_config" object(s) if "usecase" objects of type “local” are defined.

For details see section IVoiceConfig.

"cloud_command":

The configuration of the prompter cloud command.

The JSON configuration must contain of one or more "cloud_command" object(s) if "usecase" objects of type “cloud” are defined.

For details see section ICloudCommandConfig.

IUserJsonConfig - Defines a verbatim block of json that is used by VoCon High as is.

Defines a verbatim block of json that is used by VoCon High as is.

This is for example used to pass json configuration block on to VoCon Hybrid that need to configured by users of Vocon High. The Connection Info object in ASR is an example of this.

This object lists not specific attributes because any valid json object can be used here, so the number, types and names of attributes are not limited.

IPrompterConfig - The configuration of the IPrompter instance(s).

Sub types:
  • "type": "local", see ILocalPrompter
  • "type": "cloud", see ICloudPrompterConfig

The configuration of the IPrompter instance(s).

The JSON configuration must contain at least one "prompter" object. Each must be named uniquely. Each "prompter" object represents the configuration of a IPrompter instance.

Key Name Value Type Required? Documentation
"type": String, one of "local", "cloud" yes Select the sub-type.
"local": { ILocalPrompter } depends Required if and only if "type": "local".
"cloud": { ICloudPrompterConfig } depends Required if and only if "type": "cloud".
"name": String yes The reference name of the prompter instance.
"default_usecase": String no The name of the usecase that must be loaded by default at creation time of the IPrompter instance.
"default_audio_scenario": String no The name of the audio scenario the prompter activates by default while playing a prompt.
"default_root_path": Path no An absolute or relative path that is by default prepended to all file paths of extra resources (email_parser_config, otd_file, tuning_files, …) configured in any usecase.

Details:

"name":

The reference name of the prompter instance.

This property contains the name of the prompter instance and must be passed to the nuance.prompter.IPrompter.create() as name argument.

"default_usecase":

The name of the usecase that must be loaded by default at creation time of the IPrompter instance.

If omitted a usecase has to be activated at a later point at run-time with an API call.

Note that the type of the referenced usecase must match to the type of the prompter (e.g. “local” or “cloud”).

"default_audio_scenario":

The name of the audio scenario the prompter activates by default while playing a prompt.

This property enables the prompter’s ability to take over activation of the default audio scenario while playing a prompt. The property contains the name of a prompting audio scenario, that is defined separately in the audio configuration. The default audio scenario is utilized when no audio scenario is active at the moment when the prompter starts to play a prompt. In this case the prompter automatically activates the default audio scenario and keeps it activated as long as the prompt is being played (i.e. if the prompt is completed, paused or aborted, the scenario is automatically deactivated again).

"default_root_path":

An absolute or relative path that is by default prepended to all file paths of extra resources (email_parser_config, otd_file, tuning_files, …) configured in any usecase.

The relative path is considered to be in relation to the JSON configuration directory. The default_root_path is not prepended to the file path of an extra resource if one of these conditions are met:

  • the default_root_path is not configured
  • the file path of the extra resource configured in a usecase is absolute
  • a specialized root path is configured that overrides the default_root_path for a specific category of resources (e.g. email_parser_configs_root_path)

Super type: IPrompterConfig

The configuration of the local IPrompter class.

This JSON object configures the local type of an IPrompter. A local type of a prompter represents a Cerence TTS Embedded engine, which is able to convert written text into audio data with the following format:

  • “encoding”: “ENCODING_PCM_16BIT”
  • “sample_rate”: 22050
  • “samples_per_channel”: 512
  • “buffer_count”: 1

"email_parser_configs_root_path": Path no An absolute or relative path that is by default prepended to an email_parser_config file path configured in any usecase. This configuration option is intended for CDFW applications only.
"otd_files_root_path": Path no An absolute or relative path that is by default prepended to an otd_file path configured in any usecase. This configuration option is intended for CDFW applications only.
"otd_audio_files_root_path": Path no An absolute or relative path that is by default prepended to an otd_audio_files_path configured in any usecase. This configuration option is intended for CDFW applications only.
"tuning_files_root_path": Path no An absolute or relative path that is by default prepended to a tuning_files path configured in any usecase.
"samples_per_channel": int no, default value is 512 Defines the number of samples per channel.
"tts_engine_lib": String no, default value is “ve” The Text-To-Speech core engine library the prompter instance will use.
"audio_file_max_play_time": int no, default value is 1200000 Defines the maximum play time of an encoded audio file inserted to the prompt text with the TTS control sequence <ESC>\audio="path"\.

Details:

"voice_data_path":

The definition of the voice data path object.

This key contains an object, that defines paths related to a certain voice data installation. Ideally the targeted object is defined with the voice_data_path key at top-level and included here via the "uses" key.

"email_parser_configs_root_path":

An absolute or relative path that is by default prepended to an email_parser_config file path configured in any usecase. This configuration option is intended for CDFW applications only.

The relative path is considered to be in relation to the JSON configuration directory. This property overrules the default_root_path specifically for all defined email_parser_config paths. An absolute file path defined in email_parser_config is not supplemented with the email_parser_configs_root_path.

"otd_files_root_path":

An absolute or relative path that is by default prepended to an otd_file path configured in any usecase. This configuration option is intended for CDFW applications only.

The relative path is considered to be in relation to the JSON configuration directory. This property overrules the default_root_path specifically for all defined otd_file paths. An absolute file path defined in otd_file is not supplemented with the otd_files_root_path.

"otd_audio_files_root_path":

An absolute or relative path that is by default prepended to an otd_audio_files_path configured in any usecase. This configuration option is intended for CDFW applications only.

The relative path is considered to be in relation to the JSON configuration directory. This property overrules the default_root_path specifically for all defined otd_audio_files_path. An absolute file path defined in otd_audio_files_path is not supplemented with the otd_audio_files_root_path.

"tuning_files_root_path":

An absolute or relative path that is by default prepended to a tuning_files path configured in any usecase.

The relative path is considered to be in relation to the JSON configuration directory. This property overrules the default_root_path specifically for all defined tuning_files paths. An absolute file path defined in tuning_files is not supplemented with the tuning_files_root_path.

"samples_per_channel":

Defines the number of samples per channel.

This number influences the audio format of the data the prompter produces. Note that the output module that receives the prompter’s data (e.g. AudioOutput) must configure the same value in its audio format.

"tts_engine_lib":

The Text-To-Speech core engine library the prompter instance will use.

The property defines the file name of the TTS library the prompter instance will use for embedded TTS processing. The name must be in the format "ve[_major_minor]" e.g. "ve" or "ve_3_2" and must not include platform-specific prefixes (like “lib”) or suffixes (like “dll”, “so”) or file path information. Any name value that does not match that format will lead to an error.

By default the prompter uses the TTS library named "ve" which corresponds to a Cerence TTS Embedded version that is greater or equal than the latest offered Cerence TTS Embedded version, and which is typically part of a standard delivery package.

Cerence TTS Embedded versions that are older than the latest offered version are supported under following circumstances:

  • The library version must be greater or equal than 2.0 .
  • The library file name must be in the above described naming format.
  • The library file must be available at the expected location in the file system.

If it is required that a prompter instance uses an older Cerence TTS Embedded version, the relevant Cerence TTS Embedded library name must be extended with version information in the above described format. E.g. if the version of the older library is 3.2, the library must be named <prefix>ve_3_2.<suffix> and the value of the "tts_engine_lib" property must be defined identically to the library part name "ve_3_2". Note, that the prompter checks during loading of the Cerence TTS Embedded library if the passed version information matches. In case of a mismatch the creation of the prompter instance fails.

Examples showing how to use an older Cerence TTS Embedded version 3.2 on different platforms:

  • On Windows: Cerence TTS Embedded library is named ve_3_2.dll and the property defined to "tts_engine_lib": "ve_3_2"
  • On Linux: Cerence TTS Embedded library is named ve_3_2.so and the property defined to "tts_engine_lib": "ve_3_2"
  • On Android: Cerence TTS Embedded library is named libve_3_2.so and the property defined to "tts_engine_lib": "ve_3_2"
"audio_file_max_play_time":

Defines the maximum play time of an encoded audio file inserted to the prompt text with the TTS control sequence <ESC>\audio="path"\.

The time value is given in milliseconds and must be in the range 0 .. 1200000 ms. By default the value is set to the maximum (1200000 ms), which corresponds to 20 minutes of play time. This Json key affects audio files that are encoded (e.g. with Ogg/Opus, 24kHz).

Playbacks of encoded audio files with higher play times are faded out right before reaching the defined "audio_file_max_play_time".

IPrompterGlobalConfig - The configuration of prompter global settings.

The configuration of prompter global settings.

The JSON configuration may consist of a "prompter_global" object. The settings are relevant for all IPrompter instances.

Key Name Value Type Required? Documentation
"prompt_history_max_count": int no, default value is 4 The maximum count of last prompts stored in the global prompt history. This configuration option is intended for CDFW applications only.
"prompt_history_text_max_length": int no, default value is 256 The maximum length of the prompt texts that are stored in the global prompt history. This configuration option is intended for CDFW applications only.

Details:

"prompt_history_max_count":

The maximum count of last prompts stored in the global prompt history. This configuration option is intended for CDFW applications only.

All prompter instances store various details (like input texts, text formats, activated use cases, etc ..) of lately played prompts in a global prompt history. Each item of the history contains the details of one prompt. The Diagnosis run-time API provides calls to obtain the content of the history for diagnosis purposes.

This property defines the maximum count the global prompt history may store. When a new item is being added to the global prompt history while the maximum count is exceeded, the oldest item is removed before (ring buffering). Setting the value to 0 disables the global prompt history.

"prompt_history_text_max_length":

The maximum length of the prompt texts that are stored in the global prompt history. This configuration option is intended for CDFW applications only.

See also property prompt_history_max_count. This property defines the maximum length of the prompt texts, that are stored in the global prompt history. If a text exceeds the maximum length, only the first characters of the defined length are stored in the global prompt history. As an indication that a text exceeded the maximum length, an info is placed in the end of the stored prompt text (for example: “… Text too long!”).

ILocalUseCase - The configuration of the local prompter usecase.

Super type: IPrompterUseCaseConfig

The configuration of the local prompter usecase.

The local type of a prompter usecase can be activated on a local type of an IPrompter instance. The JSON configuration must contain one or more local "usecase" object(s). Each usecase name has to be unique. A usecase must at least reference a "voice_config" object, but may have additional application-specific data like exception tuning files, text normalization rules or active prompt databases.

Key Name Value Type Required? Documentation
"email_parser_config": String no The absolute or relative path to the configuration file for email preprocessor core engine. This configuration option is intended for CDFW applications only.
"tuning_files": [ { ITuningFile } ] no A list of tuning files to be loaded when activating the usecase.
"otd_file": String no The absolute or relative path to an OTD concepts file. This configuration option is intended for CDFW applications only.
"otd_audio_files_path": String no The absolute or relative path to the location where audio files are stored used in OTD concepts. This configuration option is intended for CDFW applications only.
"active_prompt_domain": String no The name of an active prompt domain.
"text_normalization_type": String no The text normalization type (TN type).
"enabled_markers": String no The selection of enabled marker types.
"voice_config": { IVoiceConfig } yes The voice configuration.

Details:

"email_parser_config":

The absolute or relative path to the configuration file for email preprocessor core engine. This configuration option is intended for CDFW applications only.

Relative path is considered to be relative to the JSON configuration directory. The content of the file is used by the email preprocessor core engine to parse XML tags that encapsulate the email content and to convert that input to the form more appropriate for audio synthesis by the text-to-speech engine.

"tuning_files":

A list of tuning files to be loaded when activating the usecase.

The list of tuningFiles specifies tuning files to be loaded when activating the usecase. The order of the list elements is crucial as per tuning file type, more recently loaded data has precedence over previously loaded data. The tuning file at list index 0 is loaded first.

Out of the above precedence rules for each tuning file type, there is an exception for prompt template, which only loads the first template it encounters, and subsequent templates will fail to be loaded. It is recommended to define at maximum one prompt template per usecase.

Note that prompt template has the highest precedence, and all user rules have precedence over all user dictionaries, and all user dictionaries have precedence over all active prompt databases.

During activation of the usecase it is checked if any tuning files are already loaded from another usecase. If yes, it is ensured that obsoleted tuning files are unloaded before start loading (obsolete may also mean that a required tuning file is already loaded but with a different mode). The prompter avoids to unload and reload required tuning files as long as the given order specified in the tuningFiles can be preserved per tuning file type. In general it is a good practice to preserve the list order of the tuningFile amongst the available usecases as much as possible to keep reloadings at a minimum.

"otd_file":

The absolute or relative path to an OTD concepts file. This configuration option is intended for CDFW applications only.

Relative path is considered to be relative to the JSON configuration directory. An OTD file contains a database holding prompter concepts. The prompter runtime API provides methods to access the database and lookup for an individual prompter concepts.

"otd_audio_files_path":

The absolute or relative path to the location where audio files are stored used in OTD concepts. This configuration option is intended for CDFW applications only.

Relative path is considered to be relative to the JSON configuration directory. A prompter concept may contain audio files names to be played. If configured, the OTD audio file path is prepended to an audio file name.

"active_prompt_domain":

The name of an active prompt domain.

It activates implicit matching for the given active prompt domain. The active prompt database(s) to load for the given domain can be achieved with the "tuning_files" list.

"text_normalization_type":

The text normalization type (TN type).

It configures the engine to normalize the text to prompt with given TN type. There are basic TN types like “spell”, “address” or “sms” as described in the Cerence TTS Embedded documentation (chapter “Guiding text normalization”). The loading of tuning files that define additional user specific TN types can be achieved with the "tuning_files" list.

"enabled_markers":

The selection of enabled marker types.

The string is a comma separated list of marker type names that need to be enabled. Possible marker types are textunit,bookmark,word,lipinfo. An enabled marker causes the prompter to send corresponding marker events to the nuance_prompter_IPromptListener interface. If nothing is defined or the string is empty, then no marker events are sent.

"voice_config":

The voice configuration.

This object contains data to configure the local TTS.

Each "usecase" object of type “local” must contain a "voice_config" object. A voice_config defines all configuration properties for a specific voice, like language, voice name, voice operating point, frequency, volume, speed, etc. A valid voice_config must define at least a name, a voice, the voice operating point and the language.

The property list below explains in short the types of the properties and possible values that can be set in the "voice_config" object. If optional properties are left out from the voice_config the default values are be assigned.

See also in the Cerence TTS Embedded documentation (description of enumeration type “VE_PARAMID” in chapter “Text-to-Speech Function Reference”) to get more details about the voice_config parameters.

(examples: "enu", "ged", "eng" etc.)

"frequency":

The sample rate.

Possible values in kHz are 8, 11, 16 or 22.

"volume":

The volume (0..100).

Volume level on a 0 to 100 scale. For each 10 points on the scale the volume changes by 3 dB.

Default value: 80

"speechrate":

The speech rate (50..400).

Speech rate level, which is a scale factor (in %) on the default speech rate of the current voice. The valid range is [50..400], with 50 having the voice speak 2x slower, and 400 having the voice speak 4x faster.

Default value: 100 (%)

"pitch":

The pitch (50..200).

Pitch level, a scale factor (in %) on the inherent pitch of the current voice. The range is [50..200]; with value 50 the voice speaks one octave lower (pitch :2), with value 200 the voice speaks one octave higher (pitch x2).

Default value: 100 (%)

"timbre":

The timbre (50..200).

Timbre level, a scale factor (in %) on the timbre of the current speaker. The range is [50..200]; For value higher than 100 [100..200] the voice sounds younger; reversely, values lesser than 100 [50..100] make the voice sound older.

Default value: 100 (%)

"waitfactor":

The wait factor (0..9).

Wait period inserted between two text units (e.g. sentences), on a scale from 0 to 9. Each unit is equivalent to 200ms of silence.

Default value: 1

"readmode":

The read mode.

Default value: READMODE_SENT, to read sentence by sentence.

" READMODE ": Describes different read modes for the TTS system.

  • "sent": Read sentence-by-sentence.
  • "char": Read character-by-character.
  • "word": Read word-by-word.
  • "line": Read line-by-line.
"initmode":

The init mode.

Default value: INITMODE_LOAD_ONCE_OPEN_ALL

" INITMODE ": Describes possible TTS initialization modes.

  • "once_open_all": Load and open all modules once.
  • "open_all_each_time": Load and open all modules for each speak request.
"textmode":

The text mode.

Text processing mode for the prompt input text passed. When setting the value to TEXTMODE_SMS, additional text processing will be enabled for better processing of SMS input.

Default value: TEXTMODE_STANDARD

" TEXTMODE ": Describes the text processing mode for the prompt input text.

  • "standard": Regular input text.
  • "sms": SMS input text.
"max_input_length":

The maximum input length.

Maximum length (in characters) for a single sentence. Values between 25 and 2500 are supported. Text fragments that are larger than this size and do not contain regular end-of-sentence punctuation, will be cut at or below this size and spoken as 2 or more separate sentences.

Default value: 250

"lidscope":

The scope for language identification.

Defines the parts of the input text for which Cerence TTS Embedded will identify the language of the text (and not expect that it’s identical to the native language of the current voice).

Default value: LIDSCOPE_USERDEFINED

" LIDSCOPE ": Defines the parts of the input text for which the Cerence TTS Embedded will identify the language of the text.

  • "none": LID is deactivated for the entire input text.
  • "userdefined": LID is activated for tagged fragments.
  • "message": LID is activated for the entire input text. IMPORTANT NOTE: The value may not be used with Cerence TTS versions >= 3.0.3 which do not support this value anymore!
"lidvoiceswitch":

The setting to enable/disable switching of language during language identification.

Defines whether Cerence TTS Embedded switches the voice when it detects foreign input, i.e. the language identification is activated and identifies the language of the text as differet from the native language of the voice.

Default value: LIDVOICESWITCH_OFF

" LIDVOICESWITCH ": Defines if the Cerence TTS Embedded has to switch the voice when it detects foreign input.

  • "off": Keep the current voice to read foreign input.
  • "on": Switch to a voice that has the foreign language as its native language.
"extraesctn":

The additional text normalization types.

Defines the additional tn types that may appear in the prompt input text. The value is a comma-separated list of types as used in <ESC>tn=<type>. If the current voice supports one or more of these additional types, it will load the language data of concern.

"extraesclang":

The comma separated list of language codes.

Defines the foreign languages that may appear in the prompt input text. The value is a comma-separated list of 3-letterlanguage codes, e.g. “eng,frf,spe,iti”. If the current voice supports one or more of these languages as foreign languages, it will load the foreign language data of concern.

"lidmode":

The mode of the language identification.

Defines the operating mode of the language identification (LID).

Default value: LIDMODE_MEMORY_BIASED

" LIDMODE ": Describes operating modes of the language identification.

  • "memory_biased": LID takes the detected language of preceding sentence into account to determine the language of the current sentence.
  • "forced_choice": LID only considers the current sentence to determine its language.
"lidlanguages":

The comma separated list of language codes.

Restricts the language identification (LID) to a subset of the supported foreign languages. The value is a comma-separated list of 3-letter language codes, e.g. “eng,frf,spe,iti”.

"final_silence_disabled":

Disables the final silence.

Disables the synthesis of the wait factor silence after the final sentence. Set to true to disable.

Note: The functionality of this key is supported with Cerence TTS Embedded versions >= 2.2.0.

Default value: false

IVoiceDataPath - The prompter voice data path.

The prompter voice data path.

The voice_data_path definition is required in combination with “local” prompter objects defining prompter instances which use the “Vocalizer Embedded” engine for text-to-speech conversion.

A voice_data_path object is used to define paths related to a voice data installation which contains a certain set of so called “Vocalizer Embedded voice packs”.

Each voice_data_path object defines a unique name, which serves as an identifier that can be used for referencing from within prompter objects.

By referencing the identical voice_data_path in multiple prompter objects, lower memory consumption and less file accesses can be achieved. E.i. when the same voice is activated on multiple prompter instances, the voice is loaded only once into the memory. Therefore the recommended way is to define one common voice_data_path referenced in all “local” prompter objects.

Key Name Value Type Required? Documentation
"name": String yes The unique name of the voice_data_path object.
"vocalizer_path": Path yes The absolute or relative path to the installation directory where all “Vocalizer Embedded voice packs” are located.
"scan_result_storage_path": Path no The absolute or relative path to a directory where the results of the scanned vocalizer_path are stored persistently.
"scan_result_storage_tag": String no The tag which is associated with the persistently stored scan results.

Details:

"name":

The unique name of the voice_data_path object.

This is used to reference it with the uses key from within prompter objects.

"vocalizer_path":

The absolute or relative path to the installation directory where all “Vocalizer Embedded voice packs” are located.

Under the vocalizer_path all common (language independent) data files and all language-/voice-specific data files (originating from the voice packs) are located.

The relative path is considered to be in relation to the JSON configuration directory.

When a prompter instance is created, the defined vocalizer_path is scanned entirely in a recursive manner in order to make an inventory of all available voice data files. During the scan all files ending with \*.hdr (so called “Vocalizer Embedded pipeline broker header files”) are accessed. If multiple prompter instances share the same voice_data_path object, the vocalizer_path is scanned one-time, when the first prompter instance is created.

Please have a look to scan_result_storage_path which can be used to store the results of the scan persistently.

"scan_result_storage_path":

The absolute or relative path to a directory where the results of the scanned vocalizer_path are stored persistently.

This key is used to reduce the time needed to create prompter instances e.g. at system start-up.

Please refer also to the description of vocalizer_path. If scan_result_storage_path is defined, the results of the scanned vocalizer_path are stored persistently into cache files that are named “<unique_prefix>_brokerdatafile.bin” and “<unique_prefix>_lwlistfile.bin” (unique_prefix relates to the name of the associated voice_data_path object). At all subsequent system start-ups the scanning of vocalizer_path is omitted and the former scan results from the cache files are used.

Note: If the voice data content under the vocalizer_path has changed, the stored cache files must be deleted manually in order to renew the cache files at next system start-up. As an alternative to avoid manual deletion of the cache files the key scan_result_storage_tag can be utilized to control renewal of the cache files.

A relative scan_result_storage_path is considered to be in relation to the JSON configuration directory.

If the defined path does not exist at initial scan, the path is automatically created. If path creation fails, this feature is treated as disabled.

"scan_result_storage_tag":

The tag which is associated with the persistently stored scan results.

This optional key is taken into use only if the scan_result_storage_path is defined. Therefore please also refer to the description of scan_result_storage_path.

Use this key to control renewal of perstistently stored scan results, e.g. when the voice data content under the vocalizer_path has changed.

At the point when the results of the scanned vocalizer_path are stored persistently into cache files in the scan_result_storage_path, the stored results are tagged with the scan_result_storage_tag value. The value of the tag is written to an additional cache file named “<unique_prefix>_tag.bin” (unique_prefix relates to the name of the associated voice_data_path object).

At subsequent system start-ups the value of scan_result_storage_tag is compared with the persistently stored tag value. In case of a mismatch, the vocalizer_path is re-scanned and the scan results in scan_result_storage_path are renewed and tagged with the new scan_result_storage_tag value.
